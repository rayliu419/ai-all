一、Agent 类型对比

从交互方式、目标定位、应用场景等维度，对比不同类型的 Agent 系统。

Coding Agent 交互式前台

代表产品：Claude Code、Cursor 等。

  • 定位：最成熟的 Agent 类型，以代码库级别的编码环境为核心
  • 主要目标：读懂复杂代码库、跑测试、修 Bug、直接开 PR 合并代码
  • 交互方式：前台交互式，在终端守着它，或者丢一个任务让它限时去改代码

企业工作流与自动化 Agent 低代码业务流程

代表产品：n8n。

  • 定位：连接业务系统，自动化工作流程
  • 主要目标：用户用自然语言描述一个业务场景（"帮我监控竞品推特，发现重要更新就总结并同步到 Slack，同时给销售发一封邮件"），Agent 自动理解意图、拆解步骤并动态调用各种 API

操作系统级别的 Agent 常驻后台

代表产品：Hermes。

  • 定位：工业级智能体框架，通用的、可自我进化的任务型 Agent
  • 特点：一个拥有长期记忆、能自己学习新技能、且可以无限分裂子智能体（Sub-agents）的分布式助理系统
  • 交互方式：后台常驻式，结合 Cron 定时器，默默挂在后台、Telegram 或服务器上 24 小时运行

LangChain vs LangGraph vs DeepAgent vs CrewAI vs Raw

框架定位优先级
LangChain 基础积木库。最全的框架，提供连接各种 LLM、数据库、内存管理和 Prompt 模板的标准化接口。实现不了复杂的逻辑，不要投入太多。 了解即可
LangGraph 带控制流的框架，支持跳转、条件执行等底层编排。最严谨的工业控制台。 P0
CrewAI 多 Agent 配合。封装了复杂的团队协作协议，快速落地原型。 P1
Raw 高端开发的选择，直接使用原生 SDK。 P0
DeepAgent 偏高层的编排，封装了底层。提供开箱即用的高级 Agent 框架。 不要投入

Raw

直接使用 Anthropic API 和 OpenAI 的 SDK。

  • 能快速使用原生 API，不用等待框架实现
  • 框架中会过度封装，自动加入上下文等，反而会引起性能问题
  • 每一行都是可解释、可调试的。适合集成到现有的后端服务中，因为不必再去适配框架复杂的异步逻辑

LangChain

Message

理解 LLM 调用时其实是每次将不同角色的 Message 作为历史一起传入：

  • SystemMessage
  • HumanMessage
  • AIMessage
  • ToolMessage

PromptTemplate

在实际开发中，你不会把给 AI 的指令死死地写在代码里，而是类似 Java 的 String.format，根据用户输入动态填充内容。

OutputParser

模型返回的通常是字符串，但你可能需要 JSON、列表或特定的 Python 对象。解析器负责将 AI 的回复"格式化"为代码可以直接处理的数据结构。是打通 MCP Tools 的关键。

LCEL

  • 仿照 Unix 的管道方式，构建 pipeline
  • 类似高阶函数，先把操作连起来，最后 invoke 的时候再开始传数据
  • 像个控制流模板，运行时做一定的参数改变就可以了

定制 Agent 类

现在很少用内置的 Agent，还是使用定制化的方式更好。LangChain 本身自带一些 Agent，但是都是黑盒。

LangGraph 重点

State

LangGraph = 节点（处理逻辑）+ 边（执行顺序）+ State（数据流动）

理解 LangGraph 的核心。State 就是一个数据流水本，记录了从开始到现在所有节点产生的数据。

想象一个城市的中心广场，所有人（节点）都会来这里：

  • 发布消息：把自己处理的结果贴到广场的公告板上
  • 查看消息：看看别人贴了什么信息，作为自己工作的输入
  • 信息汇总：所有人的工作成果都汇集在这个广场

特点

  • 可累积性：State 像滚雪球一样，越滚越大，每个节点都往上加东西
  • 可覆盖性：如果节点返回了同名字段，会覆盖旧值。可以自定义 Reduce 方式来改变默认行为
  • 节点间通信的唯一方式：在 LangGraph 中，节点之间不能直接对话，只能通过 State 交流

CrewAI 重点

能快速出活，多 Agent 协调。

三、Agent Engineer 经验总结

Agent Engineer 的核心是：对接 LLM 和 Agent 的能力。

本质：LLM 只会做"成语接龙"，Agent 是通过 LLM 的"成语接龙"能力，跟需求对应过来。

关键挑战

上下文增强：为了获取实时数据，需要调用 Tool 来获取输入的增强（传递给 LLM 的输入是用户输入 + 补充输入）。

MCP 技能：编写极其精准的工具元数据，利用 MCP 协议标准化数据输入输出。

对话久了模型会变傻，且 Token 成本极高：一次长程对话中，后续每次都会把前面所有内容都发给 LLM。窗口爆掉模型就会变傻。MCP Tool 的返回也算在历史里。

方案：动态上下文管理——向量数据库或长短期记忆总结，动态将关联度最高的信息塞入上下文。

模型在长任务中容易复读或陷入死循环。
→ 状态管理与工作流设计。

模型生成的命令可能具有破坏性，无法直接在服务器运行。
→ 搭建安全沙盒，让模型在受控环境中执行代码并获取真实反馈。

需要的技能

  • 编程硬实力：Python
  • 协议理解：深入研究 MCP，理解它是如何解耦模型和工具的
  • 框架熟练度：LangGraph（目前处理复杂逻辑的首选）或原生的函数调用逻辑
  • 数据敏锐度：知道如何通过 LangSmith 观测 Agent 的思考轨迹

从 0 到 1 的五个阶段

Agent Engineer 的工作是不断通过工程手段弥补大模型局限性的过程。

1初生期 — 指令驱动 (Prompting)

场景：你让模型写一段代码，它直接吐出一段话。

解锁技能：结构化 Prompt (Structured Prompting)

解决问题：模型输出不可控的问题

实战：学习使用 XML 标签或 JSON Schema 强制模型输出固定格式。

"你必须始终以 JSON 格式输出，包含 Thought 和 Action 两个字段。"

2成长期 — 赋予手脚 (Function Calling / MCP)

场景：模型说它想查数据库，但它没连网，也没权限。

解锁技能：工具定义与接入 (Tool Engineering & MCP)

解决问题：模型"光说不练"的问题

实战：编写一个 MCP Server。为你的 Python 函数写出完美的 description。

3进化期 — 逻辑闭环

场景：模型调用工具失败了（如 API 报错），它就卡住了或胡说八道。

解锁技能：ReAct 循环与逻辑编排 (LangGraph)

解决问题：模型"一错就死"的问题，实现自主纠错

实战：使用 LangGraph 构建一个图：

  • 节点 A (LLM)：决定动作
  • 节点 B (Tool)：执行动作
  • 判定边 (Edge)：如果执行结果包含 "Error"，指回节点 A 重新思考

4成熟期 — 记忆管理 (Memory & RAG)

场景：任务变复杂，Agent 做了 10 步之后忘记了第 1 步的目标。或需要处理 500 页说明书。

解锁技能：向量数据库 (Vector DB) 与状态保存 (Checkpointing)

解决问题：模型"贵人多忘事"的问题

实战：

  • 引入持久化存储，将之前的 Observation 存入数据库
  • 使用 RAG 技术，只把与当前步骤相关的文档片段塞进上下文

5卓越期 — 护栏与评估 (Eval & Guardrails)

场景：Agent 表现很出色，但偶尔会尝试执行 rm -rf /，或在处理财务报表时算错。

解锁技能：输入/输出过滤与自动评测 (Evals)

解决问题："不敢上线"的问题

实战：

  • 编写硬代码拦截高危操作
  • 构建 50 个典型案例库，每次修改 Prompt 或工具定义后自动跑全量测试，对比成功率
💡 小结

三种 Agent 类型从不同维度切入，覆盖了从代码开发到企业自动化再到通用智能体的完整谱系。开发时根据场景选择合适的框架（Raw / LangGraph 优先），在五个阶段的工程实践中逐步提升 Agent 的能力上限。
