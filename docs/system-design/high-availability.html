1. 多 AZ 部署 — 冗余容灾

多可用区（Availability Zone）部署是云原生高可用的第一道防线。

关键设计原则
  • N+1 容量规划：如果有 3 个 AZ，每个 AZ 的容量要能够承受总流量的 50%（即挂了一个剩下两个能扛住）。而不是刚好 33%，那样挂了一个就过载了。
  • 无状态服务优先：服务本身不存储状态，状态放在外部存储（DB、Cache），这样任何 AZ 的 Pod 都可以替代。
  • 数据层也要多 AZ：服务多 AZ 了但数据库是单 AZ，数据库挂了照样全完。
  • 测试 AZ 故障：定期做 AZ 故障演练（Chaos Engineering），确保真实出事时切换是丝滑的。
核心：冗余 + 反脆弱设计，不仅要多副本，还要保证单个副本失效后整体仍有足够容量。

2. 主从切换 / Leader Election — 自动故障转移

🎯 场景：K8s 集群只有一个 master 节点，某天 master 挂了

整个集群无法调度新 Pod、无法响应 kubectl 命令，所有运维操作瘫痪。

没有主从切换的问题：

  • 单 master 是经典单点故障。Master 挂了，已有 Pod 短时间还能跑，但不能创建新 Pod、不能扩容、不能做任何控制面操作。
  • 恢复 master 需要人工介入，可能花 30 分钟甚至更久。

更多 Leader Election 的例子

系统 选举机制 说明
Kafka Controller ZooKeeper / KRaft 负责分区 Leader 选举和 broker 上下线管理
Redis Sentinel Raft 风格选举 Sentinel 集群选出 Leader，由 Leader 执行 failover
ZooKeeper ZAB 协议（类似 Raft） Leader 处理写，Follower 处理读
Consul Raft 服务发现和配置管理
核心：通过分布式一致性协议（Raft / ZAB / Paxos）自动选举新 Leader，实现控制面故障的自动恢复。

3. 健康检查 + 自动剔除 — 快速剔除故障节点

🎯 场景：微服务集群有 10 个 Pod，其中 2 个因为内存泄漏处于半死不活状态

进程还在但响应超时。Load Balancer 继续把请求发过去，导致 20% 的请求超时。

没有健康检查的问题：

  • "进程活着" ≠ "服务正常" — 进程可能在但已经 OOM、死锁、连不上数据库
  • LB 只检查端口是否通，端口通但服务不正常的情况检测不到
  • 人工发现问题再处理，可能已经过了 10 分钟

有了完善的健康检查：

探针类型 作用
Liveness Probe 检测进程是否还活着，不活了就重启 Pod。解决"进程卡死"的问题。
Readiness Probe 检测服务是否准备好接受流量，没准备好就从 Service 的 endpoints 中剔除。
Startup Probe 给慢启动的应用一个宽限期，避免 Liveness Probe 在应用没启动完就杀掉它。

更高层面的健康检查

  • AWS ALB Target Group Health Check：ALB 定期对后端实例做 HTTP 检查，unhealthy 的自动剔除
  • Consul / Eureka 服务注册：服务实例定期向注册中心发心跳，心跳超时自动下线
  • 数据库连接池健康检查：HikariCP 等连接池定期检测连接是否有效，无效的自动丢弃
核心：多层次的健康检测（进程级 → 服务级 → 依赖级），将故障节点从负载池中快速摘除。

4. Circuit Breaker — 熔断保护

🎯 场景：订单服务调用支付服务，支付服务挂了

每次调用都要等 5 秒超时才返回。订单服务线程池 200 个线程，全部卡在等待支付超时，新的订单请求排队，最终订单服务自己也挂了。下游的故障雪崩式地拖垮了整条链路。

没有熔断的问题：

  • 级联失败（Cascading Failure）：A 调 B，B 调 C，C 挂了 → B 线程池耗尽 → A 线程池耗尽 → 系统崩溃
  • 重试风暴：每个请求超时后重试，下游本来就扛不住，还被重试流量压得更狠

有了 Circuit Breaker：

  • 借鉴电路保险丝原理。正常时电路闭合（Closed），故障太多时保险丝断开（Open），直接不再发请求，快速返回错误
  • 过一段时间尝试性放几个请求过去（Half-Open），如果成功就恢复，失败就继续断开

实际效果：支付服务挂了 → 熔断器在 10 次失败后打开 → 后续请求直接返回"支付暂不可用"（50ms），不再等 5 秒超时 → 线程池不会耗尽 → 其他功能不受影响。

常用框架

Resilience4j（Java 推荐）、Sentinel（阿里）、Envoy / Istio（Service Mesh 层面熔断）

核心：快速失败（Fail Fast）优于耗时等待，通过熔断器状态机（Closed → Open → Half-Open）在系统层面级联保护。

5. Bulkhead — 舱壁隔离

🎯 场景：一个服务同时依赖支付服务和库存服务，共用同一个 HTTP 线程池

支付服务变慢了，200 个线程全堆在等待支付响应。这时候库存查询也进不来了——明明库存服务好好的，但线程都被支付占了。

没有隔离的问题：不同下游调用共享资源（线程池、连接池、内存），一个慢的拖累所有。类比：船体没有隔离舱，一处漏水就整船沉没。

有了 Bulkhead（舱壁隔离）：给不同的下游调用分配独立的资源池。

  • 支付调用用线程池 A（50 线程），库存调用用线程池 B（50 线程）
  • 支付线程池满了？只影响支付相关功能，库存查询照常

隔离的不同粒度

隔离方式 说明 特点
线程池隔离 每个下游一个独立线程池 最常见，有超时保护
信号量隔离 用计数器限制并发数，不创建新线程 比线程池轻量，无超时保护
进程隔离 关键服务独立部署，物理隔离 最彻底但成本最高
K8s Resource Limits 给 Pod 设置 CPU/Memory limits 一个 Pod 出问题不影响同节点其他 Pod
核心：将资源池按下游/功能分区，一个分区的故障不会耗尽全局资源，实现故障隔离。

6. Retry + Exponential Backoff + Jitter — 优雅重试

🎯 场景：100 个客户端同时调用一个服务，服务短暂过载返回 503

所有客户端立刻重试 → 服务瞬间又收到 100 个请求 → 又 503 → 又全部重试……无限循环，服务永远恢复不了。这就是"重试风暴"（Retry Storm）。

没有退避策略的问题：

  • 固定间隔重试：所有客户端在同一时刻重试，形成"惊群效应"（Thundering Herd）
  • 无限重试：下游已经挂了，客户端不停重试只会加重下游负担

有了 Exponential Backoff + Jitter：

  • 指数退避：第 1 次等 1 秒，第 2 次等 2 秒，第 3 次等 4 秒……逐渐拉长重试间隔，给下游喘息时间
  • 随机抖动（Jitter）：在退避时间上加上随机偏移，100 个客户端不再同时重试，分散到不同时刻
  • 最大重试次数：超过 N 次就放弃，避免无限重试

哪些错误应该重试？

✅ 需要重试

  • 网络超时
  • 503 Service Unavailable
  • 429 Too Many Requests

❌ 不需要重试

  • 400 Bad Request
  • 401 Unauthorized
  • 404 Not Found
核心：重试要有退避策略，避免重试风暴。加上 jitter 打散重试时刻。只对可恢复的错误重试。

7. Rate Limiting + Load Shedding — 过载保护

Rate Limiting（限流）

  • 单 Pod 限流：保护单个实例不被打崩溃，即使整体集群有容量，单个 Pod 也有极限
  • 全局限流：保护下游资源（数据库、第三方 API），防止总请求量超过下游承受能力
  • 常见算法：令牌桶（Token Bucket）、漏桶（Leaky Bucket）、滑动窗口（Sliding Window）

Load Shedding（甩负荷）

  • 比限流更激进：当系统已经过载时，主动丢弃一部分请求
  • 优先保证核心请求（下单、支付），牺牲非核心请求（推荐、浏览历史）

实际例子

  • Envoy / Nginx：在网关层做限流，应用层不用管
  • AWS API Gateway：内置 throttling，每秒/每日请求限制
  • Google 的 CoDel（Controlled Delay）：根据队列等待时间动态丢弃请求。请求在队列里等太久就直接丢弃——因为等它被处理完时，客户端可能早就超时了
核心：拒绝过量请求，保护系统不因过载而崩溃。Load Shedding 在限流基础上进一步做优先级调度。

8. Graceful Degradation — 优雅降级

🎯 场景：电商 App 首页需要聚合 6 个服务的数据，推荐服务挂了

没有降级：首页直接报错，用户什么都看不到。或者一直等推荐服务超时，首页加载慢 5 秒+。

有了优雅降级：

  • 推荐服务挂了？首页显示一个"热门商品"的静态列表（预先缓存好的 fallback 数据）
  • 其他 5 个服务正常，用户完全感觉不到异常
  • 类比：飞机的一个引擎坏了，靠剩下的引擎继续飞，而不是直接坠机

降级策略的层次

优先级 策略 说明
1 返回缓存数据 用上一次成功的结果，可能有点旧但比没有好
2 返回默认值/静态数据 推荐服务挂了就展示热门商品缓存
3 隐藏功能模块 这个模块暂时不显示，不影响其他模块
4 关闭非核心功能 全站关闭评论、关闭推荐，保证核心链路（搜索→下单→支付）畅通
5 只读模式 禁止写入操作，只允许查询
核心：部分故障不导致整体不可用，通过缓存、静态数据、功能降级等手段保证核心体验。

9. Active-Active vs Active-Passive — 多活架构

🎯 场景：全球业务公司，数据中心在新加坡。日本和美国的用户每次访问延迟 200ms+

Active-Passive（主备模式）

  • 一个数据中心承担所有读写流量，另一个数据中心只做数据同步，平时不接流量
  • 优点：简单，没有数据冲突问题
  • 缺点：备站资源闲置浪费；切换时有停服窗口；备站长期不接流量，真切换时可能有各种意想不到的问题

Active-Active（多活模式）

  • 多个数据中心同时接受读写流量
  • 优点：用户就近访问延迟低；没有资源浪费；一个挂了其他无缝接管
  • 缺点：必须处理数据冲突（同一条数据被两个 DC 同时修改）；架构复杂度大幅增加

冲突处理方式

策略 说明
按用户分片 日本用户的数据只在东京写，美国用户的数据只在弗吉尼亚写。避免同一条数据在两个 DC 被修改
LWW（Last Write Wins） 时间戳最新的覆盖。简单但可能丢失数据
CRDT 数据结构层面保证自动合并，不冲突。详见 geo-replication.md

业界实践：DynamoDB Global Tables（LWW）、Cassandra 多 DC（无主天然多活）、CockroachDB（Region 内 Raft 强一致，详见 geo-replication.md）

核心：Active-Active 延迟更低、利用率更高，但必须解决数据冲突问题。按用户分片是最简单的冲突规避手段。

10. Quorum / 多数派机制 — 防脑裂

🎯 场景：3 节点集群（A、B、C），网络分区导致 A 和 B、C 断开连接

A 觉得自己想选新 leader；B、C 觉得 A 挂了想选新 leader。如果两边都接受写入 → 脑裂 → 数据分叉 → 灾难。

Quorum 的解决方式：

  • 任何写操作必须得到超过半数节点（Quorum = N/2 + 1）的确认才算成功
  • 3 节点的 Quorum = 2。A 独自一个分区只有 1 票，达不到 Quorum，自动变成只读/下线
  • B 和 C 有 2 票 ≥ Quorum，可以选出新 leader 继续服务
  • 核心思想：网络分区时，只有包含多数派的那部分分区能继续写入，少数派自动停止接受写入

使用 Quorum 的系统

Etcd/Raft、ZooKeeper/ZAB、Kafka KRaft、MongoDB（w: majority）、Cassandra（可调一致性 QUORUM）

核心：通过多数派决策防止脑裂，保证集群在网络分区时只有一个分区能提供服务，避免数据不一致。

总结

高可用的本质就是三个字：冗余（多副本）、检测（快速发现故障）、恢复（自动切换到健康节点）。上面所有模式都是这三者的不同组合。
  • 多 AZ 部署、Active-Active：基础设施层面的冗余
  • 健康检查、心跳、超时：故障检测手段
  • Leader Election、熔断、重试退避、限流降级：自动恢复和过载保护机制
  • Bulkhead、Quorum：故障隔离和一致性保障
