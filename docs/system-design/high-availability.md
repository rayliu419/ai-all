## 高可用设计模式

### 多 AZ 部署 - 冗余容灾
- **关键设计原则：**
    - N+1容量规划：如果有3个AZ，每个AZ的容量要能够承受总流量的 50%（即挂字一个剩下两个能扛住）。而不是刚好33%，那样挂了一个就过载了。
    - 无状态服务优先：服务本身不存储状态，状态放在外部存储（DB、Cache），这样任何AZ的Pod都可以替代。
    - 数据层也要多AZ：服务多AZ了但数据库是单AZ，数据库挂了照样全完。
    - 测试AZ故障：定期做AZ故障演练（Chaos Engineering），确保真实出事时切换是丝滑的。

### 主从切换 / Leader Election - 自动故障转移
- 场景：K8s集群只有一个master节点，某天master挂了，整个集群无法调度新Pod、无法响应kubectl命令，所有运维操作瘫痪。
- 没有主从切换时的问题：
    - 单master是经典单点故障。Master挂了，已有的Pod短时间还能跑，但不能创建新Pod、不能扩容、不能做任何控制面操作。
    - 恢复master需要人工介入，可能花30分钟甚至更久。
- **更多 Leader Election 的例子：**
    - **Kafka Controller**：Kafka 集群中的 Controller 负责分区 Leader 选举和 broker 上下线管理。通过 Zookeeper（旧版）或 KRaft（新版）选出唯一的 Controller。
    - **Redis Sentinel**：Sentinel 集群通过 Raft 风格选举选出 Leader，由 Leader 执行 failover。
    - **ZooKeeper**：本身使用 ZAB 协议（类似 Raft），选出 Leader 处理写，Follower 处理读。
    - **Consul**：使用 Raft 协议做 Leader election，提供服务发现和配置管理。

### 健康检查 + 自动剔除 - 快速剔除故障节点
- 场景：一个微服务集群有 10 个 Pod，其中 2 个因为内存泄漏处于半死不活状态——进程还在，但响应超时。Load Balancer 不知道它们有问题，继续把请求发过去，导致 20% 的请求超时。
- 没有健康检查时的问题：
    - “进程活着” ≠ “服务正常”。进程可能在但已经 OOM、死锁、连不上数据库。
    - LB 只检查端口是否通，端口通但服务不正常的情况检测不到。
    - 人工发现问题再处理，可能已经过了 10 分钟，用户投诉一堆了。
- 有了完善的健康检查：
    - Liveness Probe：检测进程是否还活着，不活了就重启 Pod。解决“进程卡死”的问题。
    - Readiness Probe：检测服务是否准备好接受流量，没准备好就从 Service 的 endpoints 中剔除，不再收到新请求。解决“启动中”或“依赖不可用”的问题。
    - Startup Probe：给慢启动的应用一个宽限期，避免 Liveness Probe 在应用还没启动完成时就把它杀掉。
- 更高层面的健康检查：
    - AWS ALB Target Group Health Check：ALB 定期对后端实例做 HTTP 检查，unhealthy 的自动剔除。
    - Consul / Eureka 服务注册：服务实例定期向注册中心发心跳，心跳超时自动下线。
    - 数据库连接池健康检查：HikariCP 等连接池定期检测连接是否有效，无效的自动丢弃。

### Circuit Breaker - 熔断保护
- 场景：订单服务调用支付服务。支付服务挂了，每次调用都要等 5 秒超时才返回。订单服务线程池 200 个线程，全部卡在等待支付超时，新的订单请求排队，最终订单服务自己也挂了。下一个游的故障，雪崩式地拖垮了整条链路。
- 没有熔断时的问题：
    - 级联失败（Cascading Failure）：A 调 B，B 调 C，C 挂了 -> B 线程池耗尽 -> A 线程池耗尽 -> 整个系统崩溃。
    - 重试风暴：每个请求超时后重试，下游本来就扛不住，还被重试流量压得更狠。
- 有了 Circuit Breaker：
    - 借鉴电路保险丝的概念。正常时电路闭合（Closed），电流正常通过。故障太多时保险丝断开（Open），直接不再发请求，快速返回错误。过一段时间尝试性放几个请求过去（Half-Open），如果成功就恢复，失败就继续断开。
- 实际效果：
    - 支付服务挂了 -> 熔断器在 10 次失败后打开 -> 后续请求直接返回“支付暂不可用”（50ms），不再等 5 秒超能力 -> 线程池不会耗尽 -> 其他功能不受影响。
- 常用框架：Resilience4j (Java 推荐)、Sentinel (阿里)、Envoy / Istio (Service Mesh 层面熔断)。

### Bulkhead - 舱壁隔离
- 场景：一个服务同时依赖支付服务和库存服务，共用同一个 HTTP 线程池（200 线程）。支付服务变慢了，200 个线程全堆在等待支付响应。这时候库存查询也进不来了——明明库存服务好好的，但线程都被支付占了。
- 没有隔离时的问题：
    - 不同的下游调用共享资源（线程池、连接池、内存），一个慢的拖累所有。
    - 类比：船体没有隔离舱，一处漏水就整船沉没。
- 有了 Bulkhead (舱壁隔离)：
    - 给不同的下游调用分配独立的资源池。
    - 支付调用用线程池 A（50 线程），库存调用用线程池 B（50 线程）。支付线程池满了？只影响支付相关功能，库存查询照常。
- 隔离的不同粒度：
    - 线程池隔离：最常见，每个下游一个独立线程池。
    - 信号量隔离：不创建新线程，用计数器限制并发数。比线程池轻量，但没有超时保护。
    - 进程隔离：关键服务独立部署，物理隔离。最彻底但成本最高。
    - K8s Resource Limits：给 Pod 设置 CPU/Memory limits，一个 Pod 出问题不影响同节点其他 Pod。

### Retry + Exponential Backoff + Jitter - 优雅重试
- 场景：100 个客户端同时调用一个服务，服务短暂过载返回 503。所有客户端立刻重试 -> 服务瞬间又收到 100 个请求 -> 又 503 -> 又全部重试……无限循环，服务永远恢复不了。这就是“重试风暴”（Retry Storm）。
- 没有退避策略时的问题：
    - 固定间隔重试：所有客户端在同一时刻重试，形成“惊群效应”（Thundering Herd）。
    - 无限重试：下游已经挂了，客户端不停重试只会加重下游负担。
- 有了 Exponential Backoff + Jitter：
    - 指数退避：第 1 次等 1 秒，第 2 次等 2 秒，第 3 次等 4 秒……逐渐拉长重试间隔，给下游喘息时间。
    - 随机抖动 (Jitter)：在退避时间上加上随机偏移，100 个客户端不再同时重试，分散到不同时刻。
    - 最大重试次数：超过 N 次就放弃，避免无限重试。
- 哪些错误应该重试？
    - 需要重试：网络超时、503 Service Unavailable、429 Too Many Requests
    - 不需要重试：400 Bad Request、401 Unauthorized、404 Not Found (这些重试一万次也不会好)

### Rate Limiting + Load Shedding - 过载保护
- Rate Limiting (限流)：
    - 单 Pod 限流：保护单个实例不被打崩溃，即使整体集群有容量，单个 Pod 也有极限。
    - 全局限流：保护下游资源（数据库、第三方 API），防止总请求量超过下游承受能力。
    - 常见算法：令牌桶 (Token Bucket)、漏桶 (Leaky Bucket)、滑动窗口 (Sliding Window)。
- Load Shedding (甩负荷)：
    - 比限流更激进：当系统已经过载时，主动丢弃一部分请求。
    - 优先保证核心请求（下单、支付），牺牲非核心请求（推荐、浏览历史）。
- 实际例子：
    - Envoy / Nginx: 在网关层做限流，应用层不用管。
    - AWS API Gateway: 内置 throttling，每秒/每日请求限制。
    - Google 的 CoDel (Controlled Delay): 根据队列等待时间动态丢弃请求。请求在队列里等太久就直接丢弃——因为等它被处理完时，客户端可能早就超时了。

### Graceful Degradation - 优雅降级
- 场景：电商 App 首页需要聚合 6 个服务的数据：用户信息、推荐商品、搜索热词、优惠券、消息数、浏览历史。推荐服务挂了。
- 没有降级时的做法：首页直接报错，用户什么都看不到。或者一直等推荐服务超时，首页加载慢 5 秒+。
- 有了优雅降级：
    - 推荐服务挂了？首页显示一个“热门商品”的静态列表（预先缓存好的 fallback 数据）。
    - 其他 5 个服务正常，用户完全感觉不到异常。
    - 类比：飞机的一个引擎坏了，靠剩下的引擎继续飞，而不是直接坠机。
- 降级策略的层次：
    1. 返回缓存数据：用上一次成功的结果。可能有点旧但比没有好。
    2. 返回默认值/静态数据：推荐服务挂了就展示热门商品。
    3. 隐藏功能模块：这个模块暂时不显示，不影响其他模块。
    4. 关闭非核心功能：全站关闭评论、关闭推荐，保证核心链路（搜索->下单->支付）畅通。
    5. 只读模式：禁止写入操作，只允许查询。

### Active-Active vs Active-Passive - 多活架构
- 场景：一家全球业务的公司，数据中心在新加坡。日本和美国的用户每次访问延迟 200ms+。而且如果新加坡数据中心挂了，全球用户都不可用。

#### Active-Passive (主备模式)
- 一个数据中心承担所有读写流量，另一个数据中心只做数据同步，平时不接流量。
- 优点：简单，没有数据冲突问题。
- 缺点：备站资源闲置浪费；切换时有停服窗口；备站长期不接流量，真切换时可能有各种意想不到的问题。

#### Active-Active (多活模式)
- 多个数据中心同时接受读写流量。
- 优点：用户就近访问延迟低；没有资源浪费；一个挂了其他无缝接管。
- 缺点：必须处理数据冲突（同一条数据被两个 DC 同时修改）；架构复杂度大幅增加。
- 冲突处理方式：
    - 按用户分片：日本用户的数据只在东京写，美国用户的数据只在弗吉尼亚写。避免同一条数据在两个 DC 被修改。
    - LWW (Last Write Wins)：时间戳最新的覆盖。简单但可能丢失数据。
    - CRDT：数据结构层面保证自动合并不冲突（详见 [geo-replication.md](./geo-replication.md)）。
- 业界实践：DynamoDB Global Tables (LWW)、Cassandra 多 DC（无主天然多活）、CockroachDB (Region 内 Raft 强一致，详见 [geo-replication.md](./geo-replication.md))。

### Quorum / 多数派机制 - 防脑裂
- 场景：一个 3 节点的集群（A、B、C），网络分区导致 A 和 B、C 断开连接。A 觉得自己想选新 leader；B、C 觉得 A 挂了想选新 leader。如果两边都接受写入 -> 脑裂 -> 数据分叉 -> 灾难。
- Quorum 的解决方式：
    - 任何写操作必须得到超过半数节点（Quorum = N/2 + 1）的确认才算成功。
    - 3 节点的 Quorum = 2。A 独自一个分区只有 1 票，达不到 Quorum，自动变成只读/下线。B 和 C 有 2 票 ≥ Quorum，可以选出新 leader 继续服务。
    - 核心思想：网络分区时，只有包含多数派的那部分分区能继续写入，少数派自动停止接受写入。
- 使用 Quorum 的系统：Etcd/Raft、ZooKeeper/ZAB、Kafka KRaft、MongoDB (`w: majority`)、Cassandra (可调一致性 `QUORUM`)。

## 总结
高可用的本质就是三个字一冗余（多副本）、检测（快速发现故障）、恢复（自动切换到健康节点）。上面所有模式都是这三者的不同组合。
