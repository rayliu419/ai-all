## harness engineering exp.

### 核心目标
- **LLM本身是概率模型，怎么让概率模型变成工程上更稳定？**
  - 通过对模型的优化+harness framework的共同配合，更好的完成复杂任务。

### LLM特点
- LLM的能力
  - 一个博学但记性有限的专家。
  - **重复输入当前问题能唤回记忆，提高能力。**

  - claude 擅长coding，gemini 擅长文档。
  - **不要用同一个Agent review自己的工作。**

  - 如果多步骤或者目标模糊，需要多反馈的，使用agent。

- 为模型建立系统工程
  - 单一复杂的提示词不稳定，切换模型以后可能效果就变了。
  - 引入task breakdown。
  - 引入plan+反思。
  - 引入tool。
  - 引入反馈循环到环境中。

### 单Agent vs. 多Agent 
- 看看cc是怎么用多Agent的。

####  核心维度对比看板

| 对比维度 | 单 Agent 架构 (Single-Agent) | 多 Agent 架构 (Multi-Agent) |
| :--- | :--- | :--- |
| **核心定位** | 全能型通用实体，独立全责处理。 | 专业化分工团队，协同处理复杂长链路。 |
| **上下文管理** | 单一上下文窗口，极易受无关信息污染，易触顶。 | 按角色拆分/压缩上下文，保持各节点信息纯净。 |
| **工具调用** | 面对多而杂的工具易过载，产生能力稀释与误调用。 | 工具依职责隔离，通过路由精准分发，支持 Tool Search。 |
| **执行效率** | 线性串行执行，单次调用延迟低，但广度受限。 | 天然支持并行化，牺牲单次延迟换取高吞吐与全面性。 |
| **提示词工程** | 提示词冗长，复杂任务下极易产生**提示词冲突**。 | 提示词短小精悍，任务边界清晰，职业划分明确。 |
| **架构重心** | 依赖单个大模型的微调或强推理能力。 | 依赖流程编排、路由决策与组织架构的确定性。 |

#### 多Agent超越单Agent的核心场景
- 多Agent架构的本质是用“组织架构的确定性”来对抗单个大模型在面对超长上下文、超多工具时的“随机性与无序性”。
- 规避无关上下文污染与海量内容筛选
  - **痛点（单Agent）：** 子任务（如长文检索、深度解析）产生的大量中间内容与主线任务无关，如果全部塞进单一上下文，会造成信息污染（Noise Pollution），导致模型在后续决策中“分心”或产生幻觉。
  - **优势（多Agent）：** 子任务由独立的 Sub-Agent 执行，产生的海量内容在内部进行**筛选、摘要和压缩**后，仅将高价值的“结论”或“关键因子”交付给主线 Agent，确保主线上下文的高纯净度。
- 超大信息空间的全面覆盖（以 Deep Research 为代表）
  - **覆盖度不足 vs 追求全：** 多Agent架构的核心目的通常**不是为了“快”，而是为了“全”**。
  - **信息空间大且延迟不敏感：** 当任务天然可拆分（如：同时对 10 家竞品公司的财报进行深度扫描），单体 Agent 受限于时序执行和注意力机制，很难做到全覆盖；多 Agent 架构通过**并行架构**横向展开，能同时探索多个信息空间分支，实现超高覆盖度。
- 专业化分工，消除“能力稀释”与“提示词冲突”
  - **打破工具低效魔咒：** 当AI需要使用到多个差异性极大的工具时，单 Agent 会因为工具描述（Tool Definitions）过多而导致**工具过载**，新增工具不仅不提效，反而会降低整体成功率。
  - **清晰的边界：** 多Agent将任务边界划清，每个 Agent 职业划分明确，只背负极少数专属工具。通过模糊性极低的**路由决策**进行任务分发，完美解决提示词冲突，实现专业人做专业事。

##### 何时该从单 Agent 拆分为多 Agent？
- **上下文触顶 (Context Ceiling)：**
  * 业务链路拉长，Prompt + 历史对话 + 工具返回的原始数据导致 Token 数量逼近模型上限。
  * 必须通过多Agent 进行**上下文拆分**与**上下文压缩**，各 Agent 只承载与其角色相关的生命周期数据。
- **工具过载 (Tool Overload)：**
  * 集成的外部API/插件数量过多（通常超过 10-15 个），模型开始频繁误调用、参数推断错误。 
  * 需要引入**Tool Search（工具搜索机制）**，或者将工具按领域（如财务类、代码类、数据库类）拆分给不同的子 Agent 托管。
- **天然可并行 (Natural Parallelism)：**
  * 业务流中存在大量互不依赖、可同时进行的探索性子任务。
  * 引入**并行架构**（如 Map-Reduce 模式），利用空间换深度，大幅提升系统总体的吞吐量和全面性。

#### 缺点
- 拆分为多Agent架构会带来**模型间通信开销增加、整体延迟（Latency）变长、路由节点可能模糊**等新挑战。
- 如果任务边界无法清晰划分，或者路由规则容易产生歧义，强行拆分反而会因”管理内耗”降低AI系统的整体效率。

### 多Agent本质区别（从 LLM 认知机制看）

从 LLM 的角度看，每个Agent确实有全局知识，但设计多Agent不是为了拥有不同的知识，而是为了拥有不同的”视角”和”能力”。

#### 1. 专注力与角色分工
类比人类团队：经理、工程师和设计师拥有相同的全局知识（公司的产品、文化、目标），但职责不同。工程师专注于技术实现，设计师专注于用户体验，经理专注于协调和决策。LLM 也是如此——让它同时扮演所有角色会导致思考混乱和效率低下。

- **单 Agent：** 像”全能选手”，需要自己思考”我应该扮演哪个角色？我接下来该做什么？”——这个过程低效且容易”精神分裂”。
- **多 Agent：** 每个Agent 被赋予明确的角色（通过系统提示词）。
  - “你是SRE 专家”→ LLM专注于系统指标和稳定性的思考。
  - “你是Python 开发者”→ LLM专注于代码逻辑和错误调试。
通过角色分工，每个Agent能更精准地运用其知识，避免不必要的认知负担。

#### 2. 可靠性与分步验证
复杂任务，尤其在高风险生产环境中，一步到位的解决方案不可靠。
- **单 Agent：** 容易出现”幻觉”或逻辑错误。如果生成不正确的方案，无人验证，任务直接失败。
- **多 Agent：** 引入验证和审查机制。
  - Agent A提出一个部署方案。
  - Agent B（安全审查官）审查该方案，指出潜在风险。
  - Agent C（经理）接收并协调两者意见，得出最终安全方案。
这种**内部辩论和多方验证**模式，极大提高了输出质量和可靠性。

#### 3. 可控性与流程编排
- **单 Agent：** 整个思考过程在一个黑盒里——只知输入和输出，很难调试中间环节。
- **多 Agent：** 整个流程透明——可清楚看到 Agent A 输出什么指标分析，Agent B 基于这些分析查询了哪些日志，最终决策 Agent 如何综合信息得出结论。这种透明度便于调试、优化和审计。

### 单Agent的优势

多Agent 并非万能，在某些场景下单 Agent 反而更优。

#### 1. 速度与效率
- **零通信开销：** 不需要等待其他 Agent 回复或协调流程。
- **单次调用：** 整个任务只需要一次 LLM 推理和工具调用（或一个 ReAct 循环）。
- **结果：** 决策链更短，响应时间更快，在低延迟场景中至关重要（如快速交易决策、即时客服响应）。
- **举例：** 简单健康检查只需要查询两个指标并立即做出 Pass/Fail 决定——用多 Agent 反而浪费通信和等待时间。

#### 2. 知识融合与交叉领域洞察
虽然多 Agent 强调分工，但有时不分工的融合思考才能带来最佳洞察。
- **复合知识运用：** 当任务需要同时、快速地权衡和融合多种知识时，单 Agent 更高效。例如一个 Agent 看到指标异常（SRE 知识）同时联想到代码库中的特定 Bug（Dev 知识），直接给出解决方案。
- **避免信息丢失：** 信息不需要在 Agent 间传递，减少了在交接过程中（Prompt限制、Summary丢失）造成信息失真的风险。
- **举例：** 如果 Agent 发现”延迟指标略微上升，同时日志中出现少量但全新的警告信息”，单 Agent 可能直接将两者关联为”新特性引入的次要性能瓶颈”；多Agent 则需要复杂的流程来确保两个 Agent 的信息能被决策 Agent 正确融合。

#### 3. 简单任务与低开销
对于目标明确、工具调用次数有限的简单任务，单 Agent 更经济简单。
- 易于开发和调试，整个逻辑在一个地方，Prompt 和工具调用清晰，维护简单。
- 成本低——只需为一次推理付费。多 Agent 系统可能需要 3-5 次甚至更多 LLM 调用，成本成倍增加。

## 长程任务支架设计

### 长程应用的问题
- 对于更复杂的任务，智能体容易随着时间的推移而偏离轨道。我们观察到智能体在执行此类任务时存在两种常见的故障模式：
  - 第一个问题是随着上下文窗口逐渐填满，模型在执行长时间任务时往往会失去连贯性。一些模型还会表现出“上下文焦虑”，即当它们接近自身认为的上下文极限时，就会过早地结束工作。
    - 上下文重置——完全清除上下文窗口并启动一个新的智能体，同时配合结构化的交接机制（该机制会传递前一个智能体的状态和后续步骤）——可以解决这两个问题。这与压缩不同，压缩会将对话的早期部分进行原地概括，以便同一代理能够继续处理缩短后的历史记录。虽然压缩保持了连续性，但它并不能让代理从头开始，这意味着上下文焦虑仍然存在。重置可以提供一个全新的状态，但代价是交接工件会保留足够的状态，以便下一个代理能够顺利地接手工作。
  - 第二个问题是自我评价。
    - 受到GAN的启发。将执行任务的智能体与评估任务的智能体分开，被证明是解决这一问题的有效方法。
    - 一旦有了这种外部反馈，生成器就有了可以迭代的具体依据。
    - 核心是设置对立的角色(系统提示词)和信息不对称。

### Anthropic解决方案
- https://www.anthropic.com/engineering/harness-design-long-running-apps
  - 无需人工干预即可构建完整的应用程序。
  - **Anthropic的核心思路是：与其指望一个模型在单一对话中解决所有问题，不如建立一个结构化的系统，通过Multi-agent的协作来克服AI的“遗忘”和“盲目”问题。**
- 解决方案
  - 多Agent架构 
    - 为了让 AI 能够长期稳定地工作，文章提出了一个典型的三角色模型： 
      - 规划者 (Planner) - 负责将模糊的用户需求转化为详细的 JSON 规范或功能列表。它不直接写代码，只负责制定“蓝图”。 
      - 生成者 (Generator)  - 根据规划者的规范，进行短距离的“冲刺”（Sprints），负责编写具体的代码（如 React 组件、API 路由）。 
      - 评估者 (Evaluator) - 这是最关键的角色。它像一个严厉的测试工程师，使用自动化工具（如 Playwright）运行生成的应用，并根据设计、功能、原创性
        等标准给出刻薄但客观的反馈。
  - 多Agent协作
    - 1. 协调的核心机制：**反馈循环。**
      - 协调不是通过“开会”完成的，而是通过生成的制品 (Artifacts)。
        - Planner -> Generator
          - Planner 将庞大的目标拆解成一个个小的、可执行的 Task Specs (任务说明书)。
        - Generator -> Evaluator
          - Generator 完成任务后，输出代码并将其部署到临时环境中。
        - Evaluator -> Planner/Generator
          - Evaluator运行自动化测试（如 Playwright 模拟用户点击），如果发现报错或设计不达标，它会生成一份详尽的Error Log/Feedback送回给Planner进行修正。
          - Planner根据反馈，更新todo list。
    - 2. 状态管理：共享上下文 
      - 由于AI的上下文窗口有限，让所有代理看到所有代码会导致“信息过载”。**关键的协调工具：共享文件系统/数据库。**
        - 隔离性 - 每个 Generator 只能看到与其任务相关的代码片段。
        - 统一视角 - Planner维护一个全局的 "Source of Truth"，记录当前哪些功能已完成，哪些待修复。这就像是项目的 README.md 和 TODO 列表。
    - 3. 如何防止“跑偏”？ 
      - 最精彩的部分。如果 Generator 陷入死循环或者写出了垃圾代码，协调机制会触发以下操作： 
        - 硬性重置 (Hard Resets) - 如果 Evaluator 连续三次给出低分，系统会强制回滚到上一个稳定的 Checkpoint，并让Planner重新审视路线图。
        - 强制多样性 (Forced Diversity) - 如果Generator总是尝试同一种失败的方法，系统会通过Prompt注入，强迫它“尝试一种完全不同的架构路径”。

## loop engineer
- Loop Engineering是指设计、构建和调优AI Agent 的“控制循环（Control Loop）”的工程实践。它不再追求让AI靠（Single-shot）就给出完美答案，而是给AI一个触发条件和一个可验证的目标，让AI在一个“执行 -> 观测 ->  反思 -> 调整”的密闭循环中自己跑，直到把任务彻底搞定。
- **长程任务的延伸。都是在讨论怎么能够做长程任务的方法学。**
- 意图定义 (Intent)
  - 明确定义任务的终点。不再是告诉 AI“请帮我写一个登录页面”，而是告诉它“在 /login 路由下实现一个表单，并且通过现有的登录集成测试”。
- 上下文摄取 (Context Fetching)
  - 动态地为每一次循环喂入最精准的上下文（如代码、错误日志、配置、甚至用户历史偏好）。
- 行动执行 (Action)
  - AI利用工具进行实操，比如修改文件、运行命令行（npm run test）、调用 API 或生成新的子计划。 
- 结果观测 (Observation)
  - 捕捉行动后的反馈。这包括编译错误、测试未通过的堆栈信息、Lint 检查报告或用户的实时反馈。这是闭环能否转起来的关键。
  - **例如SDUI这种完全自制的LIB，由于训练时无法学习，在没有额外的动作时，几乎无法做好。**
  - 例如模版写错了，没有办法提供一个好的错误反馈，导致AI很难观察结果(这个结果不仅仅是简单输出，而是要给出错误的细节)
  - maven或者编译器，错误的时候给出了很多细节，这样AI很容易学习。
  - **如果要做的好，要专门设计反馈机制。**
- 状态调整与反思 (Adjustment)
  - AI根据观测到的失败或成功，更新自己的内部状态和下一步计划，然后重新进入循环，直到达成最初定义的可验证目标或触发终止条件。
