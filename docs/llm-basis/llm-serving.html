Pre-training vs. Serving

可以把 Pre-training（预训练）和 Serving（推理服务）比喻为"读书"和"现场答辩"——前者是默默积累知识，后者是即时回应挑战。

维度Pre-trainingServing
核心任务 像海绵一样吸收海量文本知识（万亿 Token），学会预测下一个词 针对用户具体输入（Prompt），实时生成高质量回复
计算特征 吞吐量导向（Throughput）。追求 GPU 算力极致利用率（MFU），通常使用极大 Batch Size 延迟导向（Latency）。追求首字延迟（TTFT）低、吞吐高，Batch Size 动态变化
数据流动 双向/并行。前向传播计算 Loss，反向传播更新权重 单向/前向。仅前向传播，权重被冻结，不发生改变
硬件需求 超大规模 GPU 集群（几百到上万张卡），需要高带宽网络（InfiniBand）频繁梯度同步 单张卡或较小集群（张量并行 TP 分布在几张卡上），更看重 显存带宽（HBM）

端到端处理流程

当用户在客户端点击发送，到屏幕上逐字出现回复，服务端经历了一个完整的生命周期。主流的 Serving 框架（如 vLLM、TensorRT-LLM）都遵循以下两个核心阶段：

Tokenizer → Prefill
计算 Prompt + KV Cache → Decoding
逐 Token 生成 → EOS / 最大长度

Prefill（预填充阶段 / 计算 Prompt）

  • 服务端的 Tokenizer 将文本切分成 Token ID，然后一次性喂给模型
  • 模型计算这些输入 Token 的 Attention 键值对，并将其缓存下来 → KV Cache

Decoding（解码阶段 / 逐字生成）

  • 基于前面上下文，自回归（Autoregressive）逐个生成后续 Token，直到遇到结束符（EOS）或达到最大长度
  • 每次只输入上一步生成的单个 Token，结合之前缓存的 KV Cache，计算下一个 Token 的概率分布，采样后输出
  • 把新 Token 的 KV 写入缓存，循环往复
Decoding 阶段每次只处理一个 Token，GPU 算力吃不满——时间都花在从显存中读取模型权重和 KV Cache 上。这是 Serving 的核心瓶颈。

KV Cache

KV Cache 是 LLM Serving 性能优化的核心。理解它的来源、存储位置和复用机制，就理解了 80% 的推理优化原理。

QKV 的含义

在 Transformer 模型的每一层中，输入一个 Token，通过三个不同的线性变换矩阵（Q, K, V）计算出三个向量：

向量含义类比
Query (Q)当前词"想要寻找什么"问题发起者
Key (K)当前词"能提供什么特征"，供别的词匹配关键词索引
Value (V)当前词"包含的实际内容"，匹配成功后输出答案内容

会话中的 KV Cache 复用

场景：用户进行三轮对话，第 3 轮发问"那第一部的导演是谁？"

  1. 会话管理器（Redis 或内存数据库）将前三轮文本按角色拼接成大文本
  2. Round 1 & 2 产生的 KV Cache 未被释放，保存在服务端显存中，与 Session_ID 绑定
  3. Round 3 请求到来时，系统发现前两轮文本的 Token 已有 KV Cache → 直接命中复用，只对新增输入进行 Prefill 计算
  4. 计算完新 Prompt 的 KV 后，追加到该 Session 的 KV Cache 链表末尾，直接进入 Decoding 阶段

KV Cache 存在哪里

KV Cache 直接存储在 GPU 显存（HBM，高带宽内存）中，而非 Redis 或 CPU 内存。

为什么必须在显存里？

  • Decoding 阶段每生成一个新词，都要和之前所有词的 KV Cache 进行一次 Attention 计算
  • 这个过程是 访存密集型（Memory-bound） 的
  • 如果把 KV Cache 放在 CPU 内存，每次生成都要经过 PCIe 总线传输数据，速度慢几十倍
