核心思想

LLM 的能力：一个博学但记性有限的专家。重复输入当前问题能唤回记忆，提高能力。

Context 管理

Context 是珍贵资源。简洁且准确的 Context 能大幅提高效率和准确。对于模型可以从代码中自己获取的，不要提供。应该提供从代码中看不到的知识。

优化 Context 的方式

  • /compact 可以压缩 context
  • 任务分层：设计完以后，使用 Memory MCP 保存最后结论，下一轮对话中 load 并开始编码
  • 新任务应该单独开新 Session
  • Rewind 清理掉中间错误的上下文
  • 怎么提供 Context 看起来也很重要——例如以 3 层结构的方式注入 context

多 Agent 使用

  • 并行化提高性能
  • 单 Agent 碰上复杂问题 Context 容易超
  • 单 Agent 聚焦单任务不容易导致"头脑爆炸"

多角色混合使用

  • Claude Code 做编程，Codex 新开 Session Review
  • 多线程操作
  • Claude 擅长 Coding，Gemini 擅长文档
  • 不要用同一个 Agent Review 自己的工作

Skills vs. Agent

  • 原子化或确定性的 → 使用 Skills
  • 多步骤或目标模糊、需要多反馈的 → 使用 Agent

合理使用不同模型

  • Thinking 模型擅长复杂推理问题
  • 能不能有一种 Skill，会根据不同的复杂度，让不同的 Agent 使用不同的模型

提高确定性的方法

  • LLM 集成程序：数学计算技能不行，尤其是大数字
  • 确定性算法：如果能用确定性算法的时候，可以直接通过 Markdown 来提供一段程序执行。显式指定能降低幻觉
  • Hooks：CC 中的 Plugin 专门支持 Hooks 来 Harness，由于 Hooks 是通过程序调用的方式，保证 100% 调用

单 Agent vs. 多 Agent

看看 Claude Code 是怎么用多 Agent 的。

核心维度对比看板

对比维度 单 Agent 架构 多 Agent 架构
核心定位全能型通用实体，独立全责处理专业化分工团队，协同处理复杂长链路
上下文管理单一上下文窗口，极易受无关信息污染，易触顶按角色拆分/压缩上下文，保持各节点信息纯净

执行效率线性串行执行，单次调用延迟低，但广度受限天然支持并行化，牺牲单次延迟换取高吞吐与全面性
提示词工程提示词冗长，复杂任务下极易产生提示词冲突提示词短小精悍，任务边界清晰，职业划分明确
架构重心依赖单个大模型的微调或强推理能力依赖流程编排、路由决策与组织架构的确定性

多 Agent 超越单 Agent 的核心场景

多 Agent 架构的本质是用 "组织架构的确定性" 来对抗单个大模型在面对超长上下文、超多工具时的"随机性与无序性"。

1. 规避无关上下文污染与海量内容筛选

痛点（单 Agent）：子任务（如长文检索、深度解析）产生的大量中间内容与主线任务无关，全部塞进单一上下文会造成信息污染（Noise Pollution），导致模型后续决策"分心"或产生幻觉。

优势（多 Agent）：子任务由独立的 Sub-Agent 执行，产生的海量内容在内部进行筛选、摘要和压缩后，仅将高价值的"结论"或"关键因子"交付给主线 Agent，确保主线上下文的高纯净度。

2. 超大信息空间的全面覆盖

多 Agent 架构的核心目的通常不是为了"快"，而是为了"全"。

当任务天然可拆分（如同时对 10 家竞品公司的财报进行深度扫描），单体 Agent 受限于时序执行和注意力机制，很难做到全覆盖；多 Agent 架构通过并行架构横向展开，能同时探索多个信息空间分支，实现超高覆盖度。

3. 专业化分工，消除"能力稀释"与"提示词冲突"

打破工具低效魔咒：当 AI 需要使用多个差异性极大的工具时，单 Agent 会因为 Tool Definitions 过多而导致工具过载，新增工具不仅不提效，反而降低整体成功率。

清晰的边界：多 Agent 将任务边界划清，每个 Agent 职业划分明确，只背负极少数专属工具。通过模糊性极低的路由决策进行任务分发，完美解决提示词冲突，实现专业人做专业事。

何时该从单 Agent 拆分为多 Agent？

  • 上下文触顶：业务链路拉长，Prompt + 历史对话 + 工具返回原始数据导致 Token 数量逼近模型上限。必须通过多 Agent 进行上下文拆分与上下文压缩。
  • 工具过载：集成外部 API/插件数量过多（通常超过 10-15 个），模型开始频繁误调用、参数推断错误。需要引入 Tool Search 或将工具按领域拆分给不同子 Agent 托管。
  • 天然可并行：业务流中存在大量互不依赖、可同时进行的探索性子任务。引入并行架构（如 Map-Reduce 模式），利用空间换深度。

⚠ 缺点：多 Agent 架构会带来模型间通信开销增加、整体延迟变长、路由节点可能模糊等新挑战。如果任务边界无法清晰划分，或路由规则容易产生歧义，强行拆分反而因"管理内耗"降低系统整体效率。

多 Agent 本质区别（从 LLM 认知机制看）

从 LLM 的角度看，每个 Agent 确实有全局知识，但设计多 Agent 不是为了拥有不同的知识，而是为了拥有不同的"视角"和"能力"。

1. 专注力与角色分工

类比人类团队：经理、工程师和设计师拥有相同的全局知识（公司的产品、文化、目标），但职责不同。LLM 也是如此——让它同时扮演所有角色会导致思考混乱和效率低下。

  • 单 Agent：像"全能选手"，需要自己思考"我应该扮演哪个角色？我接下来该做什么？"——这个过程低效且容易"精神分裂"。
  • 多 Agent：每个 Agent 被赋予明确的角色（通过系统提示词）：
    • "你是 SRE 专家" → LLM 专注于系统指标和稳定性的思考
    • "你是 Python 开发者" → LLM 专注于代码逻辑和错误调试
    通过角色分工，每个 Agent 能更精准地运用其知识，避免不必要的认知负担。

2. 可靠性与分步验证

复杂任务，尤其在高风险生产环境中，一步到位的解决方案不可靠。

  • 单 Agent：容易出现"幻觉"或逻辑错误。如果生成不正确方案，无人验证，任务直接失败。
  • 多 Agent：引入验证和审查机制：
    1. Agent A 提出一个部署方案
    2. Agent B（安全审查官）审查该方案，指出潜在风险
    3. Agent C（经理）接收并协调两者意见，得出最终安全方案
    这种内部辩论和多方验证模式，极大提高了输出质量和可靠性。

3. 可控性与流程编排

  • 单 Agent：整个思考过程在一个黑盒里——只知输入和输出，很难调试中间环节。
  • 多 Agent：整个流程透明——可清楚看到 Agent A 输出什么指标分析，Agent B 基于这些分析查询了哪些日志，最终决策 Agent 如何综合信息得出结论。这种透明度便于调试、优化和审计。

单 Agent 的优势

多 Agent 并非万能，在某些场景下单 Agent 反而更优。

1. 速度与效率

  • 零通信开销：不需要等待其他 Agent 回复或协调流程
  • 单次调用：整个任务只需要一次 LLM 推理和工具调用（或一个 ReAct 循环）
  • 结果：决策链更短，响应时间更快，在低延迟场景中至关重要（如快速交易决策、即时客服响应）

举例：简单健康检查只需要查询两个指标并立即做出 Pass/Fail 决定——用多 Agent 反而浪费通信和等待时间。

2. 知识融合与交叉领域洞察

虽然多 Agent 强调分工，但有时不分工的融合思考才能带来最佳洞察。

  • 复合知识运用：当任务需要同时、快速地权衡和融合多种知识时，单 Agent 更高效。例如一个 Agent 看到指标异常（SRE 知识）同时联想到代码库中的特定 Bug（Dev 知识），直接给出解决方案。
  • 避免信息丢失：信息不需要在 Agent 间传递，减少了在交接过程中（Prompt 限制、Summary 丢失）造成信息失真的风险。

举例：如果 Agent 发现"延迟指标略微上升，同时日志中出现少量但全新的警告信息"，单 Agent 可能直接将两者关联为"新特性引入的次要性能瓶颈"；多 Agent 则需要复杂的流程来确保两个 Agent 的信息能被决策 Agent 正确融合。

3. 简单任务与低开销

对于目标明确、工具调用次数有限的简单任务，单 Agent 更经济简单。

  • 易于开发和调试，整个逻辑在一个地方，Prompt 和工具调用清晰，维护简单
  • 成本低——只需为一次推理付费。多 Agent 系统可能需要 3-5 次甚至更多 LLM 调用，成本成倍增加

长程任务支架设计

长程应用的问题

对于更复杂的任务，智能体容易随着时间的推移而偏离轨道。两种常见的故障模式：

1. 上下文焦虑与丢失连贯性

随着上下文窗口逐渐填满，模型在执行长时间任务时往往会失去连贯性。一些模型还会表现出"上下文焦虑"——当接近自身认为的上下文极限时，过早结束工作。

方案：上下文重置——清除上下文窗口并启动新 Agent，配合结构化的交接机制（传递前一个 Agent 的状态和后续步骤）。

这与压缩不同：压缩会原地概括对话早期部分；重置提供全新状态，但代价是交接工件。重置可以解决上下文焦虑，因为 Agent 可以从头开始。

2. 自我评价

受到 GAN 的启发。将执行任务的 Agent 与评估任务的 Agent 分开，被证明是解决这一问题的有效方法。

一旦有了外部反馈，生成器就有了可以迭代的具体依据。

核心是设置对立角色（系统提示词）和信息不对称。

经验

Loop Engineering：想靠一轮完整的提示词达到效果是不现实的。关键是构建长程任务获得良好结果：提供 AI 能读懂的反馈。

防止错误 propagate

  • 错误积累的问题：让整个流程中每个 Task 不能积累
  • Checkable Contracts
  • Blocking Gates
  • Feedback from Failed Checks

Agent Loop

  • 定制化 MCP Tool：参数校验 + 错误信息 Surface
  • Agent Loop 的自我校验就像人的交互式校验
  • 模型胡说八道时，记录 Tool 调用的结果，跟结论交互校验
  • 如果不符合，加入这轮质疑的反馈，保证 LLM 纠错
  • 系统本身对 AI 的友好性

Anthropic 解决方案

参考：Harness Design for Long-Running Apps

核心思路：与其指望一个模型在单一对话中解决所有问题，不如建立一个结构化的系统，通过 Multi-Agent 的协作来克服 AI 的"遗忘"和"盲目"问题。

三角色模型

规划者 (Planner) — 将模糊用户需求转化为详细 JSON 规范或功能列表。不直接写代码，只负责制定"蓝图"。

生成者 (Generator) — 根据规范进行短距离"冲刺"（Sprints），编写具体代码。

评估者 (Evaluator) — 最关键的角色。使用自动化工具运行生成的应用，根据设计、功能、原创性等标准给出反馈。

多 Agent 协作

协调核心机制：反馈循环。协调不是通过"开会"完成的，而是通过生成的制品。

  • Planner → Generator：将目标拆解成小的、可执行的 Task Specs
  • Generator → Evaluator：完成任务后，输出代码并部署到临时环境
  • Evaluator → Planner/Generator：运行自动化测试，发现报错或不达标，生成 Error Log/Feedback 送回 Planner 修正

状态管理：共享上下文

关键协调工具：共享文件系统/数据库。

  • 隔离性：每个 Generator 只能看到与其任务相关的代码片段
  • 统一视角：Planner 维护全局 "Source of Truth"，记录哪些功能已完成，哪些待修复

防止"跑偏"

  • 硬性重置：Evaluator 连续三次低分 → 强制回滚到上一个稳定 Checkpoint，Planner 重新审视路线图
  • 强制多样性：Generator 总尝试同一种失败方法 → 通过 Prompt 注入强迫它"尝试一种完全不同的架构路径"

Loop Engineering

Loop Engineering 是指设计、构建和调优 AI Agent "控制循环 (Control Loop)" 的工程实践。不再追求让 AI 靠一次性 (Single-shot) 给出完美答案，而是给 AI 一个触发条件和一个可验证的目标，让它在一个"执行 → 观测 → 反思 → 调整"的密闭循环中直到彻底完成任务。

长程任务的延伸。都在讨论怎么做长程任务的方法学。

循环要素

意图定义 (Intent) — 明确定义任务终点。不再告诉 AI"请帮我写一个登录页面"，而是"在 /login 路由下实现一个表单，通过现有的登录集成测试"。

上下文摄取 (Context Fetching) — 动态注入最精准的上下文（代码、错误日志、配置、用户历史偏好等）

行动执行 (Action) — AI 利用工具实操：修改文件、运行命令行、调用 API 或生成新的子计划

结果观测 (Observation) — 捕捉行动后的反馈（编译错误、测试堆栈、Lint 报告、用户反馈）。这是闭环能否转起来的关键

状态调整与反思 (Adjustment) — AI 根据观测结果更新内部状态和下一步计划，重新进入循环

反馈设计的关键性

  • SDUI 这种完全自制的 LIB，由于训练时无法学习，在没有额外动作时几乎无法做好
  • 模板写错了，没有提供好的错误反馈，导致 AI 很难观察结果
  • Maven 或编译器在错误时给出了很多细节，这样 AI 很容易学习
  • 如果要做得好，要专门设计反馈机制
💡 总结

Harness Engineering 的核心是把概率性的 LLM 包装在确定性的工程框架中。通过 Context 管理、多 Agent 协作、反馈循环设计、Loop Engineering 等手段，让 LLM 的不可靠性被系统性地约束和弥补。
