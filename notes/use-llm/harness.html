LLM 是概率模型，怎么让概率模型在工程上更稳定？
通过对模型的优化 + Harness Framework 的共同配合，更好地完成复杂任务。

核心思想

LLM 的能力：一个博学但记性有限的专家。重复输入当前问题能唤回记忆，提高能力。

围绕 AI 构建 Task 的 Infra

先使用 AI 来构建某个 Task 的 Infra。例如对于 Agent Chatbot，把下载执行记录、配置 Emux MCP、Benchmark 都搭建出来，会极大提高效率。

标准模板

模板很重要。Coding Agent 从 Few-Shot 学习，会被不好的模板带偏。

Context 管理

Context 是珍贵资源。简洁且准确的 Context 能大幅提高效率和准确。对于模型可以从代码中自己获取的，不要提供。应该提供从代码中看不到的知识。

优化 Context 的方式

  • /compact 可以压缩 context
  • 任务分层：设计完以后，使用 Memory MCP 保存最后结论，下一轮对话中 load 并开始编码
  • 新任务应该单独开新 Session
  • Rewind 清理掉中间错误的上下文
  • 怎么提供 Context 看起来也很重要——例如以 3 层结构的方式注入 context

多 Agent 使用

  • 并行化提高性能
  • 单 Agent 碰上复杂问题 Context 容易超
  • 单 Agent 聚焦单任务不容易导致"头脑爆炸"

多角色混合使用

  • Claude Code 做编程，Codex 新开 Session Review
  • 多线程操作
  • Claude 擅长 Coding，Gemini 擅长文档
  • 不要用同一个 Agent Review 自己的工作

Skills vs. Agent

  • 原子化或确定性的 → 使用 Skills
  • 多步骤或目标模糊、需要多反馈的 → 使用 Agent

合理使用不同模型

  • Thinking 模型擅长复杂推理问题
  • 能不能有一种 Skill，会根据不同的复杂度，让不同的 Agent 使用不同的模型

提高确定性的方法

  • LLM 集成程序：数学计算技能不行，尤其是大数字
  • 确定性算法：如果能用确定性算法的时候，可以直接通过 Markdown 来提供一段程序执行。显式指定能降低幻觉
  • Hooks：CC 中的 Plugin 专门支持 Hooks 来 Harness，由于 Hooks 是通过程序调用的方式，保证 100% 调用

为模型建立系统工程

单一复杂的提示词不稳定，切换模型以后效果可能就变了。

  • 引入 Task Breakdown
  • 引入 Plan + 反思
  • 引入 Tool
  • 引入反馈循环到环境中（如 UT、集成测试 + 错误输出）
  • 对立的 Agent 也是一种反馈循环

长程任务支架设计

长程应用的问题

对于更复杂的任务，智能体容易随着时间的推移而偏离轨道。两种常见的故障模式：

1. 上下文焦虑与丢失连贯性

随着上下文窗口逐渐填满，模型在执行长时间任务时往往会失去连贯性。一些模型还会表现出"上下文焦虑"——当接近自身认为的上下文极限时，过早结束工作。

方案：上下文重置——清除上下文窗口并启动新 Agent，配合结构化的交接机制（传递前一个 Agent 的状态和后续步骤）。

这与压缩不同：压缩会原地概括对话早期部分；重置提供全新状态，但代价是交接工件。重置可以解决上下文焦虑，因为 Agent 可以从头开始。

2. 自我评价

受到 GAN 的启发。将执行任务的 Agent 与评估任务的 Agent 分开，被证明是解决这一问题的有效方法。

一旦有了外部反馈，生成器就有了可以迭代的具体依据。

核心是设置对立角色（系统提示词）和信息不对称。

经验

Loop Engineering：想靠一轮完整的提示词达到效果是不现实的。关键是构建长程任务获得良好结果：提供 AI 能读懂的反馈。

防止错误 propagate

  • 错误积累的问题：让整个流程中每个 Task 不能积累
  • Checkable Contracts
  • Blocking Gates
  • Feedback from Failed Checks

Agent Loop

  • 定制化 MCP Tool：参数校验 + 错误信息 Surface
  • Agent Loop 的自我校验就像人的交互式校验
  • 模型胡说八道时，记录 Tool 调用的结果，跟结论交互校验
  • 如果不符合，加入这轮质疑的反馈，保证 LLM 纠错
  • 系统本身对 AI 的友好性

Anthropic 解决方案

参考：Harness Design for Long-Running Apps

核心思路：与其指望一个模型在单一对话中解决所有问题，不如建立一个结构化的系统，通过 Multi-Agent 的协作来克服 AI 的"遗忘"和"盲目"问题。

三角色模型

规划者 (Planner) — 将模糊用户需求转化为详细 JSON 规范或功能列表。不直接写代码，只负责制定"蓝图"。

生成者 (Generator) — 根据规范进行短距离"冲刺"（Sprints），编写具体代码。

评估者 (Evaluator) — 最关键的角色。使用自动化工具运行生成的应用，根据设计、功能、原创性等标准给出反馈。

多 Agent 协作

协调核心机制：反馈循环。协调不是通过"开会"完成的，而是通过生成的制品。

  • Planner → Generator：将目标拆解成小的、可执行的 Task Specs
  • Generator → Evaluator：完成任务后，输出代码并部署到临时环境
  • Evaluator → Planner/Generator：运行自动化测试，发现报错或不达标，生成 Error Log/Feedback 送回 Planner 修正

状态管理：共享上下文

关键协调工具：共享文件系统/数据库。

  • 隔离性：每个 Generator 只能看到与其任务相关的代码片段
  • 统一视角：Planner 维护全局 "Source of Truth"，记录哪些功能已完成，哪些待修复

防止"跑偏"

  • 硬性重置：Evaluator 连续三次低分 → 强制回滚到上一个稳定 Checkpoint，Planner 重新审视路线图
  • 强制多样性：Generator 总尝试同一种失败方法 → 通过 Prompt 注入强迫它"尝试一种完全不同的架构路径"

Loop Engineering

Loop Engineering 是指设计、构建和调优 AI Agent "控制循环 (Control Loop)" 的工程实践。不再追求让 AI 靠一次性 (Single-shot) 给出完美答案，而是给 AI 一个触发条件和一个可验证的目标，让它在一个"执行 → 观测 → 反思 → 调整"的密闭循环中直到彻底完成任务。

长程任务的延伸。都在讨论怎么做长程任务的方法学。

循环要素

意图定义 (Intent) — 明确定义任务终点。不再告诉 AI"请帮我写一个登录页面"，而是"在 /login 路由下实现一个表单，通过现有的登录集成测试"。

上下文摄取 (Context Fetching) — 动态注入最精准的上下文（代码、错误日志、配置、用户历史偏好等）

行动执行 (Action) — AI 利用工具实操：修改文件、运行命令行、调用 API 或生成新的子计划

结果观测 (Observation) — 捕捉行动后的反馈（编译错误、测试堆栈、Lint 报告、用户反馈）。这是闭环能否转起来的关键

状态调整与反思 (Adjustment) — AI 根据观测结果更新内部状态和下一步计划，重新进入循环

反馈设计的关键性

  • SDUI 这种完全自制的 LIB，由于训练时无法学习，在没有额外动作时几乎无法做好
  • 模板写错了，没有提供好的错误反馈，导致 AI 很难观察结果
  • Maven 或编译器在错误时给出了很多细节，这样 AI 很容易学习
  • 如果要做得好，要专门设计反馈机制
💡 总结

Harness Engineering 的核心是把概率性的 LLM 包装在确定性的工程框架中。通过 Context 管理、多 Agent 协作、反馈循环设计、Loop Engineering 等手段，让 LLM 的不可靠性被系统性地约束和弥补。
