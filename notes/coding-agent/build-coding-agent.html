如何构建一个 Coding Agent

Claude Code 或是类似的顶级 Coding Agent 之所以强大，并不仅仅是因为底层的 LLM 聪明，更在于一套围绕代码逻辑、工具链和工作流的系统性工程。

📖 目录
1. 针对代码的底层模型微调（Post-training） 2. 环境感知与工具集成（Tool Use） 3. 检索与上下文管理（RAG for Code） 4. 专有的交互协议（MCP & Specialized Protocols） 5. 软件工程思维的编排（Orchestration）

1. 针对代码的底层模型微调（Post-training）

基础模型虽然读过 GitHub，但并不代表它懂"如何修 Bug"。

长文本上下文（Context Window）

Agent 需要理解整个代码库（Repo-level context），能让它"看到"项目全局。

指令遵循与推理强化

通过 RLHF 专门针对代码逻辑进行训练，让模型学会"思考步骤"（Chain of Thought），例如在写代码前先写计划。

工具调用优化

专门训练模型输出高度精准的 JSON 或特定格式来调用 Shell、读取文件或执行测试。

2. 环境感知与工具集成（Tool Use）

Agent 和普通 Chatbot 的最大区别在于它有"手"。

感知能力

它能自主运行 ls、grep、find 来探索陌生的目录结构，而不是等着你把代码粘贴给它。

执行环境

在沙盒化的终端（Sandbox）里运行编译器、Linter 和测试框架（如 Pytest、Jest）。

自愈能力（Self-healing）

当它写出有 Bug 的代码时，它能看到编译器报错，并根据报错信息自动循环修复，直到测试通过。

3. 检索与上下文管理（RAG for Code）

面对几十万行代码，模型塞不下所有内容，必须有精准的"记忆提取"。

代码索引（Indexing）

利用符号图（Symbol Graph）或抽象语法树（AST）来理解类与函数之间的调用关系，而不仅仅是文本匹配。

相关性排序

当你问"登录逻辑在哪"时，系统要能跨文件提取相关的 auth.py、User.java 和数据库配置。

4. 专有的交互协议（MCP & Specialized Protocols）

Claude Code 背后使用了 MCP（Model Context Protocol）协议。

标准化标准

MCP 允许模型以统一的方式访问本地文件、数据库和 API。

流式编辑

传统的模型会重写整个文件，而优秀的 Agent 采用类似 git patch 的方式，只修改受影响的代码行，这既节省了 Token，又降低了出错率。

5. 软件工程思维的编排（Orchestration）

这是最核心的"灵魂"：让模型像程序员一样思考。

阶段 传统 Chat Coding Agent（Claude Code）
理解 只看当前对话 扫描文件树，搜寻关键词
规划 直接吐代码 生成 Plan.md，确认修改范围
执行 让你自己复制粘贴 自动执行 sed 或文件写入命令
验证 说"应该没问题" 运行 npm test，看绿灯才停
