为什么需要 Memory

大模型的运作模式是每次对话独立计算，需要将前面所有的历史一起发送。新的 Session 不包含以前的内容，因此老的 Session 中的背景知识，新 Session 完全不知道。

  • 跨会话记忆：记忆能让 Agent 跨越时间和 Session（对话阶段），记住关于你的核心信息。
  • 提高对话效率：无需重复介绍背景。
  • 支撑复杂任务流：现实世界中的任务很少能通过"一问一答"解决，往往需要拆解成几小时甚至几天的工作流。例如执行自动化爬虫并分析数据时，Agent 需要记住第一步拿到了哪些 URL，第二步报错了需要重试哪一部分，第三步该合并哪个表格。
  • 驱动反思与成长：人类通过复盘来成长，Agent 也需要通过记忆完成反思（Reflection）。没有记忆，Agent 就会陷入在同一个坑里反复跌倒的死循环。

Memory 系统的难点

核心难点在于：在开始做某个任务之前，如何把以前相关的知识准确获取出来，以更好地完成新的任务。

  1. 检索的精准度与高噪音（Retrieval Precision vs. Noise）——向量相似度（如余弦相似度）只管"长得像"，不管"对不对"。检索出来的历史碎片可能包含大量无关信息，严重污染 LLM 的上下文，导致智能体判断失误。
  2. 上下文窗口的"能放下"≠"能用好"——虽然 Gemini 等模型的上下文窗口已扩展到非常大，但大窗口不等于高效使用。
  3. 记忆更新极难平衡——哪些需要永久固化（如用户的名字、核心偏好），哪些应随时间淡化？缺乏科学的衰减算法（Decay Mechanism）会导致记忆库无限膨胀。
  4. 原始日志不足以支撑智慧——如果仅仅是把聊天记录原封不动地存下来，Agent 的智商会非常受限。

当前的主要设计模式

混合检索：向量 + 关键词 + 知识图谱

  • 向量检索看语义相似度——让 Agent 知道哪些词"经常在一起出现"（相关性）。
  • 关键词检索（FTS5）看精准匹配——找到包含特定函数名或术语的历史。
  • 知识图谱看因果和实体关系——让 Agent 明白"什么是原因，什么是结果"（因果性），从而具备真正的逻辑推理与反思能力。
  • 散落的聊天记录只是信息噪音；提炼为点与线、实体与关系的图谱后，记忆才凝聚成高密度、可随时检索的"知识网络"。向量记忆带来了感性的"直觉"，知识图谱注入了理性的"骨架"。

知识图谱的更多优势

  • 多跳推理（Multi-hop Reasoning）：向量数据库无法遍历关系链。例如"哪些供应商同时给A公司和B公司的竞争对手供货？"——向量搜索从结构上就不可能完成这种查询。知识图谱通过显式的实体-关系边，支持确定性的多跳遍历。
  • 时序理解（Temporal Reasoning）：知识图谱支持双时间模型（bi-temporal），事实不会被删除而是标记为"过期"，实现完整的时光回溯查询。Zep/Graphiti 在时序推理任务上比纯向量方案高出 18.5% 的准确率。
  • 可解释性（Explainability）：向量检索的相似度分数是黑盒的，无法解释"为什么像"；知识图谱的每个答案都可以追溯到具体的节点和边路径，满足审计合规要求。
  • 冲突事实处理：向量数据库将每次输入视为不可变事实；当事实改变时，旧文本仍然存在，检索到冲突信息会混淆 LLM。知识图谱可以将旧事实标记为 invalid_at，在保留历史的同时保持当前事实清晰。
  • 全局整体理解（Global Query）：在跨数据集的整体性问题（如"这个语料库的核心主题是什么？"）上，标准向量 RAG 在某些基准测试中准确率为 0%，而 GraphRAG 通过社区检测和层次化摘要大幅领先。

各方案详解

claude-mem 旁路观察全自动

github.com/thedotmack/claude-mem

架构

Worker Service（HTTP server on port 37777）+ SQLite 数据库 + AI 观察者模型。

写入流程

  1. 用户对话 → Claude Code 使用工具
  2. Hook 触发 → Worker Service 接收
  3. AI 观察者模型分析 → 写入 SQLite observations 表
  • 会话元数据 + 观察记录 + 每轮对话总结
  • 图结构记录 Memory
  • 异步调用小模型来总结（避免阻塞）

读取流程

  • SessionStart hook — 启动时注入上下文（基于项目 + 时间倒序 + 类型/概念过滤）
  • MCP Server — 对话过程中提供 ChromaDB 向量搜索，基于当前 prompt 的语义相似度检索历史

Claude Code 原生方案

分为会话内上下文管理（In-Session）和跨会话持久化记忆（Cross-Session / Persistent Memory）两大核心维度。

  • 会话内上下文管理：压缩过往对话、保留关键上下文
  • 跨会话持久化（CLAUDE.md / MEMORY.md）：以文件形式持久化，工具直接读写
  • CLAUDE.md 就是每次 Session 启动时自动注入的 Memory 之一

详见：memory-system-analysis.html

Hermes Agent 无损存储FTS5

长效环境背景

存储活动项目、技术栈、核心依赖关系等长期信息。

原始对话无损保存

所有 CLI 终端和消息网关（Telegram/Discord/Slack）的原始对话均保存于本地的 ~/.hermes/state.db。

  • 使用 SQLite FTS5（全文检索）技术实现精准关键词匹配
  • 应用场景：长对话截断后查找几周前的某句代码或特定 Bug → 直接检索原始文本，而非依赖 LLM 模糊总结

Memory Tool（Agent 自维护）

通过 add（添加）、replace（替换/合并）和 remove（删除）三个原子操作管理 USER.md 和 MEMORY.md。

MemTensor / MemOS 层次化自进化

层次化存储（Hierarchical Storage）

层级内容特点
L1 轨迹层 — Traces 纯原始的对话和工具执行日志 完整保留，不做提炼
L2 策略层 — Policies "用户喜欢什么样的代码风格"、"这个 Bug 应该怎么修"等规则 Agent 自己总结的抽象规则
L3 世界模型 — World Models 项目的核心架构、不可变的事实 高稳定性，极少变动

混合检索引擎（Hybrid Retrieval）

  • 向量检索（Vector Search）：模糊的、语义相关的记忆（"找一下之前聊过的关于高并发优化的地方"）
  • 全文检索（SQLite FTS5）：精准的关键词匹配（"查找包含 auth_v2_final 这个函数名的历史对话"）
  • 智能去重（Deduplication），确保用最少 Token 带来最精准的背景

异步吞吐 & MemScheduler

Agent 在前台流畅对话，内存的写入、向量化（Embedding）、合并与擦除都在后台异步完成，毫秒级延迟。

自进化与纠错

  • 记忆整合（Consolidation）：多条零散记忆指向同一件事时，自动融合成一条高信息密度记忆
  • 反思纠错（Feedback Loop）：人类说"不对，刚才那个方案不安全，以后别用了"→ 主动定位旧记忆，修改、补充或无痕替换

动态 Memory 与 Cache 的冲突

动态 Memory 会在对话中插入 Token（相当于在对话中动态插入了额外内容）。如果设计不当，动态更新的 Memory 会高频触发 Cache Miss（缓存未命中），导致 API 费用暴涨、首字延迟飙升。

如何让 Memory 与 Cache 完美共存：Static-to-Dynamic Layout

严格控制 Prompt 拓扑结构，将最不容易变动的内容放在最前面，将最高频变动的内容放在最后面：

Prompt =
  System Prompt + Tool Schemas   {绝对固定 (100% Hit)
  Chat History (Append Only)     线性增长 (自动续期缓存)
  Dynamic Memory / Timestamp     置于末尾 (仅此处 Miss)
