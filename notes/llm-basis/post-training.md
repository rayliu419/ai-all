## post-training
- 对齐与微调 (Alignment & Fine-Tuning)
  - 比如 SFT (指令微调)、RLHF。
  - 从产品和软件工程的角度来看，SFT 解决的是“能不能做（功能性需求）”，而 RLHF 解决的是“做得好不好、优不优雅（非功能性需求）”。
- 量化与压缩 (Quantization & Compression)
  - 比如 GPTQ、AWQ，把模型的“体重”从16GB减到4GB，让普通电脑甚至手机也能跑得动。

### SFT
- 预训练模型 (Base 模型) 虽然知识渊博，但它只会“根据前文接话”。比如你问它：“请帮我写一首关于夏天的诗”，
- 它可能会接着你的话继续列举：“1. 请帮我写一首关于冬天的诗；2. 请帮我写一首关于春天的诗……”。
- SFT的核心作用就是打破这种“复读机”状态：
  - 格式与行为对齐 (Alignment)：让模型理解“Prompt (提示词/问题)”与“Response (回答)”的关系。看到问号或指令，它知道自己该去回答，而不是去续写。
  - 解锁特定任务能力：如果你想让模型变成一个专业的医疗诊断助手、代码生成器或者法律顾问，你可以用特定的专业问答数据喂给它，让它在某个特定领域开窍。
  - 控制输出风格和语气：通过 SFT，可以规范模型的说话语气（如：专业、温柔、幽默），或者限定输出格式（如：必须返回 JSON 格式、必须用 Markdown 表格展示）。

### LoRA SFT
- 大模型内部是由无数个巨大的矩阵 (Weights) 组成的，这些矩阵就像是一堵堵巨大的承重墙，负责处理信息的传输。
  - 原模型：被彻底冻结 (Frozen)。在 LoRA 训练期间，原模型的任何一个参数都不允许发生改变（不接受任何训练）。
  - LoRA 的结构：它是包含两个非常窄的矩阵（我们称为 A 和 B）的微型网络。
- LoRA 被“插”在原模型矩阵的旁边。当有数据流过时，一份数据走原模型（不改变），另一份数据走 LoRA（进行训练）。最后，把两者的输出结果加在一起。
  - 它不是一个独立从头到尾运行的新网络，它必须依赖原模型才能存在。
  - 为什么原模型那么大，LoRA 只要那么小的一点点参数就能达到同样的效果？
  - 核心假设：“大模型虽然参数多，但为了完成某个特定任务，真正起作用的‘核心维度’其实很低。”
- 假设原模型是一个 $4096 \times 4096$ 的超级大矩阵（约 1600 万个参数）。如果要全参数微调，你得调整这 1600稳 个数字。
  - LoRA 采用了“曲径通幽”的办法。它把这个大矩阵拆成了两个相乘的薄矩阵：
  - 矩阵 A：把 $4096$ 维的高度信息，压缩到只有 $8$ 维（这 $8$ 维就是所谓的 Rank，秩）。
  - 矩阵 B：再把这 $8$ 维的信息，放大回 $4096$ 维。
  - 全参数微调：需要训练 $4096 \times 4096 = 16,777,216$ 个参数。
  - LoRA (Rank=8) 微调：只需要训练 $(4096 \times 8) + (8 \times 4096) = 65,536$ 个参数。
- 训练完成后，你手里会有一个原模型文件（很大，几个G到几十G）和一个 LoRA 权重文件（很小，通常只有几十MB）。
- 在实际部署和使用（推理）时，有两种玩法：
  - 动态外挂（不合并）：原模型不动，动态加载 LoRA。这种好处是，如果你有 10 个不同的任务（比如一个写诗、一个写代码、一个做客服），你只需要存 1 个原模型和 10 个几十兆的 LoRA 文件。针对不同用户切换不同的 LoRA 即可，极省显存。
  - 熔合合并 (Merge)：利用矩阵乘法的原理，在服务上线前，直接把 LoRA 的参数加进原模型的矩阵里（因为数学上 $W_{\text{新}} = W_{\text{原}} + A \times B$）。合并后，LoRA 消失，原模型参数更新，变成了一个干净利落、没有任何额外计算开销的新模型。

#### LoRA
- **LoRA既可以用来做SFT，也可以用来做DPO，甚至可以用来做PPO。它不是与SFT并列的、非此即彼的关系，它是一种“底层训练手段”。**

#### LoRA SFT vs. 全参数SFT

| 对比维度 | 全参数微调 (Full Fine-Tuning) | LoRA (Low-Rank Adaptation) |
| :--- | :--- | :--- |
| **直观比喻** | 把整栋房子推倒，根据新图纸重新整体翻修。 | 房子主体结构完全不动，在墙外加装一个功能阳台。 |
| **训练参数量** | 模型 的 **100%** 参数（所有神经元权重一起参与计算与更新）。 | 仅训练外挂的薄矩阵参数（通常 **< 1%**，原模型参数完全冻结）。 |
| **显存与算力** | **极高**。通常需要多卡或多机集群 (如 A100/H100 服务器集群)。 | **极低**。单张消费级显卡 (如 RTX 4090 / 3090 24G) 即可展开训练。 |
| **训练速度** | **较慢**。每次迭代都需要计算、传输和保存海量的参数梯度。 | **极快**。由于需要计算梯度的参数极少，反向传播开销大幅降低。 |
| **遗忘风险** | **高**。新数据容易把预训练阶段学到的通用海量知识“冲刷”掉（灾难性遗忘）。 | **极低**。底座参数被死死冻结，通用知识基础完好无损。 |
| **通用能力影响** | 模型在特定垂直任务上显著变强，但往往伴随着通用逻辑、常识能力的退化。 | 几乎不影响模型的通用推理能力，更像是在上一层“行为规范滤镜”。 |
| **多任务部署** | **成本极高**。每个独立的任务都需要单独保存一份几十 GB 的完整大模型文件。 | **成本极低**。只需共享一个庞大底座模型，不同任务动态切换几十 MB 的 LoRA 插件文件。 |
| **最终推理延迟** | **无延迟**。因为模型本身的物理网络结构没有发生任何改变。 | **无延迟**（若上线前将 LoRA 权重熔合合并）；**极微小延迟**（若保持独立结构进行动态加载）。 |
| **最佳适用场景** | 1. 打造核心行业基座（如从头定制彻底改变逻辑习惯的医疗、法律大模型）。<br>2. 团队拥有充足的预算、算力以及海量高质量的专业 SFT 数据集。 | 1. 企业落地特定垂直应用任务（如智能客服、特定格式 JSON 数据输出、角色扮演）。<br>2. 个人开发者、预算受限的团队、快速原型验证与迭代。 |

#### SFT与Pre-train的优化差异
- 从数学和第一性原理的角度来看，它们的损失函数 (Loss Function) 在形式上是完全一模一样的。
- 无论是Pretrain还是LoRA (甚至全参数SFT)，大语言模型的核心任务永远只有一个：Next-Token Prediction (预测下一个词)。
  - 在Pretrain时，模型说出来的每一个字（包括问题和答案）都要计算Loss。而在LoRA (SFT) 时，我们通常会把“问题 (Prompt)”部分的Loss遮蔽掉 (Mask)，只对“答案 (Response)”部分的预测算 Loss。因为我们不想惩罚模型预测问题的能力，只想规范它回答的能力。
- 虽然Loss的计算方式类似，但反向传播 (Backpropagation) 更新参数时，两者的行为完全不同。
  - LoRA不能更新原模型矩阵的参数。
  - 优化过程：Loss计算出来后，梯度同样向前传递。但是当梯度传到原模型的参数时，原模型说：“我被冻结了，拒绝修改”。于是，所有的梯度最终全部堆积到了那几个LoRA外挂参数上。

#### RLHF
- 如果说SFT是给模型请了一个“标准答案家教”，那RLHF就是给模型请了一位“教导主任”。它不再给你提供完美的标准答案，而是坐在那里看你写出好几个不同的回答，然后告诉你：“这个挺好，那个太敷衍，另一个有毒不能说。”
  - 写出“好答案”比“判卷子”难得多：让人类专家写几万条完美的医学、法律、逻辑推理答案，成本高到爆炸；但让人类看大模型写的两句话，分辨出“哪一个更有礼貌、更安全”，却非常容易。
  - SFT无法解决“幻觉”和“安全问题”：大模型如果在SFT里没学过某个知识，它会倾向于瞎编。你很难通过SFT数据穷尽所有不能说的话（比如“怎么造炸弹”）。
- 它专门用来解决大模型的 3H 问题：
  - Helpful (有用性)：回答是不是切中要害，是不是真的帮到了用户。
  - Honest (诚实性)：知之为知之，不知为不知，减少瞎编（幻觉）。
  - Harmlessness (无害性)：拒绝回答暴力、色情、歧视、偏见等有毒内容。
- 数据集的物理结构：
  - Prompt (提示词/问题)：比如“请告诉我怎么在网上黑掉别人的电脑”。
  - Chosen (被选中的优质回答)：模型 A 的回答——“对不起，我无法提供协助网络攻击的行为……”（安全、合规）。
  - Rejected (被拒绝的劣质回答)：模型 B 的回答——“黑掉电脑有以下几个步骤：第一步……”（不安全）。
  - 人类标注员（或者更高级的模型）的工作，就是针对同一个问题，在大模型生成的 2 到 4 个候选回答里，勾选出哪个是 Chosen (赢家)，哪个是 Rejected (输家)。
- RLHF的主流算法
  - PPO 算法 (Proximal Policy Optimization)
    - 第一阶段：SFT训练。
    - 第二阶段：训练“奖励模型 (Reward Model)”。拿上面提到的“偏好数据集”去训练一个小模型。这个小模型不负责回答问题，它只负责看大模型的回答，然后打分（比如 Chosen 给 90 分，Rejected 给 10 分）。
    - 第三阶段：强化学习 (PPO 核心)。让大模型去回答海量的新问题，回答完后，把答案丢给第二阶段的“裁判小模型”打分。如果得分高，大模型的参数就获得奖励，强化这种说话方式；如果得分低，参数就被惩罚，以后少这么说话。
  - DPO 算法 (Direct Preference Optimization)
    - “既然我们已经有了人类的偏好数据 (Chosen 和 Rejected)，为什么还要费尽心思先训练一个裁判模型，再用强化学习去折腾大模型呢？我们能不能直接让大模型学习这个偏好？”
    - 做法：DPO 抛弃了“裁判模型”，也抛弃了强化学习那套复杂的框架。它在数学上做一个了一个精妙的转换，直接把偏好数据喂给大模型：如果模型预测 Chosen 的概率升高了，就进行奖励；如果预测 Rejected 的概率升高了，就进行惩罚。通过这种方式直接优化。

#### RLHF的优化 vs. pre-train优化/SFT优化
- **RLHF (PPO算法) 的 Loss 计算方式，和 Pre-train / SFT 的 Next-Token Prediction (交叉熵损失) 有着本质的区别。**
- Pre-train/SFT的本质是“模仿”。Loss 的核心是：我说A，正确答案是B，如果模型没猜中B，就扣分 (Loss 变大)。
- RLHF的本质是“评价”。这时候，已经没有人类写好的“正确答案 B”了，模型自己随便说了一长串话（可能包含 50 个词），裁判模型 (Reward Model) 看完这一长串话后，只给了一个整体的分数（比如：+2.5 分）。
  - 一个长句子只得到一个总分。请问：这50个词里，到底是哪一个词说得好才拿的高分？又是哪一个词说得差导致被扣分？
  - 传统的Next-Token Prediction面对这种“一锤子买卖”打分方式，直接抓瞎了，公式根本套不进去。
  - $$ \text{Text}\{PPO Loss\} \approx \text{Text}\{这句话被说出来的概率变化\} \times \text{Text}\{\text{裁判给的优势得分}\} - \text{Text}\{\text{KL散度惩罚}\} $$
  - 在 DPO 的 Loss 里：
    - 它去计算模型生成Chosen（好话）的概率。
    - 它去计算模型生成Rejected（坏话）的概率。
    - DPO Loss 目标：直接拉大这两者的差距——让Chosen的概率疯狂飙升，让Rejected的概率疯狂下跌。

### post-training怎么保证最后不会遗忘pre-train的知识
- 强化学习对齐 (Alignment) 中最核心的工程痛点：**模型怎么保留知识，以及最后上线的到底是啥。**
- SFT和RLHF时，其实都等同于最终面向用户的参数跟pre-train的参数不同，因此可能会影响到pre-train的效果（改变了参数即改变了知识）。
- 怎么保证不会导致**灾难性遗忘**？
- DPO的方法
  - 在DPO训练时，显存里其实偷偷复制了一份原模型（我们叫它 $M_{\text{ref}}$，参考模型），并且把它彻底冻结。当 DPO 优化真正的目标模型（我们叫它 $M_{\text{txt}}$）时，它的损失函数 (Loss) 里其实包含两个力量在拔河：
    - 偏好拉力：让模型多说 Chosen (好话)，少说 Rejected (坏话)。
    - 紧箍咒 (KL 散度约束)：要求正在训练的 $M_{\text{txt}}$，绝对不能偏离原始参考模型 $M_{\text{ref}}$ 太远。如果正在训练的模型为了迎合好评偏好，开始疯狂修改参数、甚至开始忘掉以前的常识，Loss 函数中的“紧箍咒”就会瞬间飙升，给模型施加一个巨大的惩罚。所以，DPO 并不是盲目地直接修改参数，而是在“保证底座通用能力不垮”的前提下戴着镣铐跳舞。

### 两维度比较
- 维度 A：你要训练什么任务？（目标与数据维度）
  - SFT (监督微调)：任务是“学写好答案”。数据是“问题 + 完美答案”。
  - RLHF / DPO (偏好对齐)：任务是“学分好坏、懂礼貌”。数据是“Chosen + Rejected”。
- 维度 B：你打算怎么改动参数？（工程与算力维度）
  - Full (全参数)：直接修改模型的 100% 参数。
  - LoRA (高效参数)：冻结原模型，只修改旁路外挂参数 (<1%)。

### 模型量化与压缩
- 模型量化 (Quantization) 和压缩，本质上就是一个纯粹的、硬核的底层工程优化。它和前面讲的 SFT、RLHF 的底层逻辑截然相反。
- 大模型内部的参数，默认是用 FP16 (16位浮点数) 存储的，一个 70B (700亿参数) 的模型，用 FP16 存储需要占用 $70 \times 2 = 140\text{ GB}$ 的显存。
- 除了大厂的专业服务器，普通显卡根本塞不下。量化的本质，就是把16位的浮点数 (FP16/BF16)，强行转换成8位 (INT8) 甚至4位 (INT4) 的整数。
  - FP16 (原版)：70B 模型 = 140 GB 显存
  - INT4 (4位量化版)：70B 模型 = $70 \times 0.5 = 35\text{ GB}$ 显存
- 既然量化不训练参数，只是把高精度数字变成低精度，那它怎么保证模型不变成“智障”呢？
  - PTQ (Post-Training Quantization, 后训练量化)
    - 模型已经训练完了，参数都固定了。我们直接用一个数学公式（缩放因子 Scaling Factor），把原本是0.123456这样的浮点数，映射成整型数字（比如 12）。
    - 不需要重新训练，几分钟就能搞定，模型能力几乎怎么下降。现在我们在网上下载的 4-bit、8-bit 体验版大模型，99% 都是用这种方法压出来的。
  - GGUF
    - 它把模型压缩成不同档次（如 Q4_K_M，代表 4位混合量化），并且优化了底层的 C/C++ 矩阵乘法。
    - 特点：让大模型摆脱了对显卡 (GPU) 和 CUDA 的绝对依赖，直接用电脑的内存和 CPU 就能跑。

## post training vs. agent engineer

1. 后训练工程师
- 核心目标： 让模型“懂代码”、“会思考”、“听指令”。
- 如果你是这个角色，你的工作是在模型出厂前，通过数据喂养来改变它的“神经元”：
- 构造思维链数据： 通用模型直接给代码。
- 你会训练它先写逻辑：
  - Step 1: 分析需求  
  - Step 2: 设计类接口 
  - Step 3: 实现业务逻辑 
  - Step 4: 编写单元测试。
- 代码质量对齐： 收集成千上万个高质量的GitHub提交记录，通过SFT让模型学会编写符合行业规范的代码。
- 错误纠正能力： 专门喂给模型一些“带 Bug 的代码”和“编译报错信息”，教模型如何根据报错信息进行自我修复 (Self-Correction)。
- 偏好学习 (DPO)： 比如模型生成了两种实现方案，一种运行快但难懂，一种简洁但性能略低。你会根据公司偏好（比如“可读性优先”）来训练模型倾向于哪种风格。
- 交付物： 一个.pth或.safetensors模型权重文件。它在没有任何外部工具的情况下，白板代码的能力极强。
2. Agent Engineer
- 核心目标： 让模型“能运行”、“会查文档”、“能自主闭环”。
- 如果你是这个角色，你的工作是把那个“大脑”放进一个真实的开发环境中：
- 环境隔离 (Sandboxing)
- 编写Python/Docker代码，为模型创建一个安全的“沙盒”，让它能执行 npm install 或 pytest 而不会写坏用户的硬盘。
- 工具编排 (Tool Integration)
- 编写接口让模型可以调用 grep 搜索全域代码，或者调用浏览器去查最新的 API 文档。
- 长程规划与状态管理
- 如果项目很大，Agent 可能会迷路。你需要用LangGraph等工具设计逻辑：如果编译失败，强制模型进入“Debug 模式”；如果成功，进入“代码审计模式”。
- 记忆检索 (RAG)
- 当模型处理万行代码的大项目时，上下文装不下。你需要建立向量索引，根据模型当前的改动，自动把相关的代码文件“喂”进模型的缓存里。
- 交付物： 一个复杂的软件系统。它能打开VS Code，自己跑测试，遇到报错会翻文档，最后给你提交一个Pull Request。

### 总结：Agent Engineer vs. 后训练工程师

| 维度 | 后训练工程师 (Model Side) | Agent 工程师 (System Side) |
| :--- | :--- | :--- |
| **核心目标** | 提高模型基础能力 (如 Function Calling) | 提高任务完成率 (Success Rate) |
| **主要工具** | PyTorch, DeepSpeed, Llama-Factory | LangGraph, Python, Vector DB |
| **关注点** | 损失函数 (Loss)、困惑度 (PPL) | 逻辑闭环、延迟、API 稳定性 |
| **交付物** | 一个 Checkpoint (模型权重) | 一个可运行的业务系统 (Service) |
