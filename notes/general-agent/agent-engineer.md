## agent engineer 
- 其核心应该是：对接LLM和Agent的能力的对接问题。
- 本质上的LLM只会做成语接龙，所以Agent是通过LLM的成语接龙能力，跟需求对应过来。
  - 上下文增强。
    - 为了获取实时的数据，需要调用Tool来获取输入的增强(及传递给LLM的输入，实际上是用户的输入+补充的输入)。
  - MCP相关的技能。编写极其精准的工具元数据，利用MCP协议标准化数据输入输出。
- 对话久了模型会变傻，且 Token 成本极高。
  - 本质上，在一次长程对话中，后续的每次对话都会把前面所有的内容都一起发给LLM。如果窗口爆掉了，模型就会变傻。
  - 类似MCP的Tool的返回，也算在历史里。
  - 怎么不发所有的，又让模型记住这些历史？动态上下文管理。
  - 向量数据库或长短期记忆总结，动态地将关联度最高的信息塞入上下文。
- 模型在长任务中容易复读或陷入死循环。
  - 状态管理与工作流设计。
- 模型生成的命令可能具有破坏性，且无法直接在服务器运行。
  - 搭建安全沙盒，让模型在受控环境中执行代码并获取真实反馈。

## 需要技能
  - 编程硬实力： Python
  - 协议理解： 深入研究MCP，理解它是如何解耦模型和工具的。
  - 框架熟练度： LangGraph (目前处理复杂逻辑的首选) 或原生的函数调用逻辑。
  - 数据敏锐度： 知道如何通过LangSmith观测Agent的思考轨迹。
   
## 从0到1的Agent engineer
- 不断通过工程手段弥补大模型局限性的过程。
- 第一阶段：初生期——指令驱动 (Prompting)
  - 场景： 你让模型写一段代码，它直接吐出一段话。
  - 解锁技能： 结构化 Prompt (Structured Prompting)。
  - 解决问题： 解决模型输出不可控的问题。
  - 实战动作： 学习使用 XML 标签或 JSON Schema 强制模型输出固定格式。
    - “你必须始终以 JSON 格式输出，包含 Thought 和 Action 两个字段。”

- 第二阶段：成长期——赋予手脚 (Function Calling / MCP)
  - 场景： 模型说它想查数据库，但它没连网，也没权限。
  - 解锁技能： 工具定义与接入 (Tool Engineering & MCP)。
  - 解决问题： 解决模型“光说不练”的问题。
  - 实战动作： 编写一个 MCP Server。为你的 Python 函数写出完美的 description。

- 第三阶段：进化期——逻辑闭环
- 场景： 模型调用工具失败了（比如 API 报错），它就卡在那儿了，或者直接胡说八道。
  - 解锁技能： ReAct 循环与逻辑编排 (LangGraph)。
  - 解决问题： 解决模型“一错就死”的问题，实现自主纠错。
  - 实战动作： 使用 LangGraph 构建一个图：
  - 节点 A (LLM)： 决定动作。
  - 节点 B (Tool)： 执行动作。
  - 判定边 (Edge)： 如果执行结果包含 "Error"，指回节点 A 重新思考。

- 第四阶段：成熟期——记忆管理 (Memory & RAG)
  - 场景： 任务变复杂了，Agent做了10步之后忘记了第1步的目标。或者需要处理一个500页的说明书。
  - 解锁技能： 向量数据库 (Vector DB) 与状态保存 (Checkpointing)。
  - 解决问题： 解决模型“贵人多忘事”的问题。
  - 实战动作： 引入持久化存储，将之前的 Observation 存入数据库。
    - 使用RAG技术，只把与当前步骤相关的文档片段塞进上下文。

- 第五阶段：卓越期——护栏与评估 (Eval & Guardrails)
  - 场景： Agent 表现很出色，但偶尔会尝试执行 rm -rf /，或者在处理财务报表时算错。
  - 解锁技能： 输入/输出过滤与自动评测 (Evals)。
  - 解决问题： 解决“不敢上线”的问题。
  - 实战动作： 编写硬代码拦截高危操作。
    - 构建50个典型案例库，每当你修改了Prompt或工具定义，自动跑一遍全量测试，对比成功率。
