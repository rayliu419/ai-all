# Agent 核心技术深度面试题

## 1. ReAct 框架的局限性 → 进阶规划方案

### ReAct 的核心逻辑

Thought → Action → Observation 的循环。

**局限性：**

- **贪心算法，缺乏全局规划。** 每一步只根据当前 Observation 决定下一步 Action，没有对最终目标的全局视野。
- **试错成本高且易漂移。** 遇到复杂任务时容易在多步之后偏离原始目标，且中间的错误步骤消耗了真实的 API 调用 / Token 成本。

### 三种进阶方案

#### Plan-Then-Act（预规划）

将规划阶段与执行阶段完全解耦。模型先生成一个完整的 Plan，然后再依次执行。

- **优点：** 结构清晰，能够进行全局资源调度，执行速度快。
- **缺点：** 脆弱性高。一旦执行过程中遇到意外情况（如 API 报错），初始规划往往失效，缺乏动态调整能力。

#### ReAct + 轻规划

在 ReAct 的循环中加入了一个"中程规划器"。它不一次性定死所有步骤，但在每一步 Action 前会维护一个动态更新的任务队列。

- **优点：** 兼顾了 ReAct 的灵活性和 Plan-Then-Act 的方向感。它会定期重新评估剩余任务，具有一定的容错性。
- **缺点：** 依然是单线推理，无法处理需要"多选一"或"回溯"的高难度逻辑难题。

#### Tree/Graph Planning（树/图规划）

引入搜索算法。模型不再只维护一条路径，而是同时探索多个可能的推理分支，并根据评估结果选择最优路径或进行回溯。

- **回溯能力：** 发现当前路径不通时，可以回到上一个节点重新选择。
- **自我评估：** 通过评估函数对中间状态打分。
- **缺点：** 极其昂贵。由于需要探索多个分支，Token 消耗量呈指数级增长，延迟非常高。

### 对比总结

| 方案 | 规划方式 | 灵活性 | 容错性 | Token 成本 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **ReAct（原始）** | 每步即时反应 | 高 | 低（易漂移） | 中 | 简单工具调用 |
| **Plan-Then-Act** | 一次性全局规划 | 低 | 低（脆弱） | 低 | 确定性强的流程 |
| **ReAct + 轻规划** | 动态任务队列 | 中高 | 中 | 中 | 多数生产环境 Agent |
| **Tree/Graph Planning** | 多分支搜索 + 回溯 | 极高 | 高 | 极高 | 数学证明、复杂推理 |

## 2. 思维链（CoT）与规划（Planning）的区别

### 核心区别

- **CoT（Chain-of-Thought）** 是让大模型逐步参与将一个复杂问题分解为一步一步的子问题并依次进行求解的过程。这些推理的中间步骤被称为思维链。
  - 核心是**"线性推理"**，侧重于将复杂问题分解为连续的子步骤，步步为营地推导答案。
  - 常用于数学、逻辑推理。
  - 本质是**语言表达的逻辑一致性**。更像是一种"边说边想"的联想过程。

- **Planning** 的核心是**"结构化设计"**，侧重于在行动前制定全局蓝图，将目标分解为可执行的任务序列，包含对行动的评估和调整。
  - 常用于 Agent 处理环境交互任务。
  - 本质是**系统执行的流程控制**。

> CoT 仅仅是"将推理过程写出来"，而 Planning 是生成"可执行的任务表"。

### 具体例子 —— 做饭

**CoT 的表现：** 像是一个厨师在自言自语

> "首先我得看看冰箱里有什么……哦，有西红柿和鸡蛋。那我得先把西红柿洗了，然后再切开。接着我得打鸡蛋，放点盐。最后生火炒菜。"

这是一种**线性独白**，没有并行、没有异常处理。

**Planning 的表现：** 像是一份标准作业程序 (Execution Plan)

- **环境检查：** 确认食材（西红柿、鸡蛋、盐）。
- **并行处理：**
  - 任务 A：洗并切西红柿。
  - 任务 B：打蛋并调味。
- **前置条件：** 只有 A 和 B 完成，才能进入任务 C（下锅）。
- **异常处理：** 若缺少盐，则触发"购买"或"替代"子程序。

## 3. 多步工具调用的鲁棒规划机制

以"调研三篇关于 RAG+RL 的论文并输出中文总结"为例。

### 分情况的错误处理

**超时 → 指数退避：**

- 第 1 次重试：等待 1s
- 第 2 次重试：等待 2s
- 第 3 次重试：等待 4s
- 超过最大重试次数 → 标记该数据源为不可用，切换到备用源。

**格式错误 → 反馈给模型：**

- 不要静默丢弃。将原始返回 + 解析错误信息一并反馈给 LLM，让模型自行判断是重试、修正解析逻辑还是跳过。
- 例如：`"API 返回了 HTML 而非预期的 JSON。原始响应前 200 字符：... 请决定：1) 重试 2) 换用备用 API 3) 跳过此步"`

**下载失败 → 多数据源：**

- 每篇论文预设 2-3 个数据源（arXiv API、Semantic Scholar、本地缓存）。
- 主源失败自动 fallback。

### 状态机 + Checkpointer

- 每个步骤执行前后都记录状态快照（checkpoint）。
- 当某个步骤失败时，能回滚到上一个正常 checkpoint，而不是从头开始。
- 实现方式：LangGraph 的 `checkpointer`，或自定义状态机。

### 动态重规划（Replan）

- 当失败次数超过阈值，不只是重试当前步骤，而是触发全局 replan。
- 将已完成步骤的结果 + 失败信息 + 剩余目标一并送入 LLM，生成新的执行计划。
- 这形成了**"执行 → 失败 → 反思 → 重规划"**的闭环。

## 4. Tree-of-Thoughts / LATS 的工作原理

### ToT（Tree-of-Thoughts）

1. **分解：** 将问题分解为多个思考步骤。
2. **生成：** 在每个步骤生成多个候选思路（分支）。
3. **评估：** 对每个候选思路打分（可以由 LLM 自己评估，也可以由独立评估器）。
4. **搜索：** 使用 BFS / DFS 选择最有希望的路径继续探索。
5. **回溯：** 当某条路径评分过低时，回到上一个分支点尝试其他方向。

### LATS（Language Agent Tree Search）

在 ToT 基础上加入**外部环境反馈**：

- 不仅依赖 LLM 的自我评估，还将真实环境反馈（如代码执行结果、API 返回值）纳入节点评分。
- 使用 MCTS（蒙特卡洛树搜索）在行动空间中搜索，平衡探索（exploration）与利用（exploitation）。

### 与线性规划相比的优势

- **突破局部最优：** 线性推理容易陷入"局部最优"。当 LLM 写到一半发现逻辑不通时，线性模式只能强行圆场，而 ToT/LATS 可以直接"剪枝"，回到上一个分叉点重新开始，从而更有可能找到全局最优解。
- **自我审视：** ToT 强制 LLM 停下来审视自己的想法，这在处理需要高度严谨性的任务时至关重要。
- **闭环实验：** 通过集成外部环境的反馈（如运行代码报错），它能像人类工程师一样根据错误日志调整策略，这种闭环实验能力是线性 Prompt 无法企及的。

## 5. 推理断层与结果偏离 —— 解决方案

"推理断层"指模型在多步推理中跳过关键逻辑步骤，"结果偏离"指执行路径逐渐远离原始目标。

### 提示工程

- **结构化中间输出：** 要求模型在每个关键步骤输出结构化状态（如 `{"step": 3, "goal_checksum": "...", "findings": [...]}`），强制其"显式确认"当前进度。
- **反事实提示：** "如果你的下一步行动无法直接推进目标，请解释原因并重新规划。"
- **目标锚定：** 在每轮 prompt 的开头重复原始目标，防止上下文漂移。

### 记忆机制

- **目标向量持久化：** 将初始目标编码为向量，在每步执行前计算与当前状态的语义相似度。若偏离超过阈值，触发 replan。
- **关键决策日志：** 将每一步的"为什么选择这个 Action"记录下来，供后续步骤回溯审计。

### 架构设计

- **双模型架构（Actor + Critic）：** Actor 负责执行，Critic 负责在每一步后检查"这一步是否在向目标靠近"。Critic 可以是一个更轻量、更快的模型。
- **周期性同步（Heartbeat Check）：** 每隔 N 步，强制将迄今为止的所有执行轨迹 + 原始目标送入 LLM 做一次"全局一致性检查"。

## 6. Agent 长期记忆模块设计

### 存储结构

**向量数据库（如 Milvus、Pinecone、Chroma）：**

- 适合存储**语义记忆**：用户偏好、对话摘要、知识片段。
- 按语义相似度检索，适合模糊匹配。

**图数据库（如 Neo4j、Neptune）：**

- 适合存储**关系记忆**：实体之间的关联（"用户 A 喜欢工具 X"、"项目 P 依赖库 L"）。
- 支持复杂的关系查询和推理。

**混合存储（推荐）：**

- 向量库处理"相似性"查询，图库处理"关联性"查询。
- 例如：用户问"上次那个跟我一起调试 PyTorch 内存问题的同事叫什么？" → 图库按关系路径检索。

### 更新策略

**记忆合并（Memory Consolidation）：**

- 类似人脑的睡眠整理。定期（如每天）将碎片化的短期记忆压缩为结构化摘要，存入长期记忆。
- 合并时做去重和冲突解决（同一实体的多条记忆取最新或加权平均）。

**遗忘机制：**

- **时间衰减：** 为每条记忆附加时间戳，检索时按 recency 加权。
- **访问频率衰减：** 长期未被检索的记忆自动降权。
- **重要性评分：** LLM 在写入记忆时给予 1-10 的重要性分数。高重要性记忆衰减更慢。
- **容量上限 + LRU 逐出。**

### 检索机制

**混合检索 + 重排序（Rerank）：**

1. 第一轮：向量检索 + 关键词检索并行，各自召回 top-20。
2. 合并去重。
3. 第二轮：用 Cross-Encoder Reranker 对候选集精排，取 top-5 注入上下文。

**分层检索：**

- 优先检索"高重要性 + 近期"记忆。
- 若结果不足，再扩大到全量记忆。

## 7. 上下文窗口溢出 —— 记忆压缩与检索策略

当历史对话远超模型上下文窗口时的应对方案：

### 滑动窗口（Sliding Window）

- 只保留最近 K 轮对话。
- **优点：** 实现简单，延迟低。
- **缺点：** 早期关键信息永久丢失（如用户一开始提到的核心需求）。
- **适用：** 实时对话、对历史依赖不强的场景。

### 总结压缩（Summarization）

- 定期对历史对话做递进式摘要。
- 常见模式：**递进式摘要链**——每次生成新摘要时，将上一轮摘要 + 新一轮对话一起送入 LLM 做合并摘要。
- **优点：** 保留了历史信息的"语义精华"，Token 占用可控。
- **缺点：** 细节必然丢失。摘要的粒度很难把握——太粗丢失关键信息，太细压缩效果差。
- **适用：** 长期用户对话、客服场景。

### 向量检索（Vector Retrieval）

- 将历史对话按片段 embedding 后存入向量库。每轮新对话时检索最相关的前 K 条历史片段。
- **优点：** 按需加载，不浪费上下文；可以保留大量历史。
- **缺点：** 可能漏掉语义相似度低但逻辑上相关的信息；检索质量高度依赖 embedding 模型和 chunk 策略。
- **适用：** 长期记忆、大规模对话历史。

### 混合方案（推荐）

```
滑动窗口（最近 N 轮原文）
+ 递进式摘要（中期压缩）
+ 向量检索（远期记忆按需召回）
```

三层信息以不同权重注入上下文，平衡新鲜度、语义完整性和关键细节。

## 8. 混合检索（Hybrid Search）

### 什么是混合检索

混合检索是指同时使用**向量语义检索**和**关键词检索（如 BM25）**，并将两者的结果进行融合排序。

### 为什么纯向量检索不够

- **向量检索擅长语义近似，不擅长精确匹配。**
- 典型失败案例：
  - 搜索 "Transformer 论文" → 向量可能返回很多关于 Transformer 架构的文章，但不一定包含 Vaswani et al. 2017 原论文。
  - 搜索 "Error code E1003" → 向量可能返回各种错误处理的文档，但找不到 E1003 的具体定义。
  - 搜索 "API v2 deprecation date" → 这个短语中每个词都很关键，向量的模糊性反而成了劣势。
- 对于**专有名词、编号、代码标识符、日期**等，精确关键词匹配远优于向量语义搜索。

### BM25 的优势

- 基于词频和逆文档频率的经典检索算法。
- 对**精确术语匹配**极其有效。
- 不需要训练，计算成本极低。

### 业务场景举例

**场景：企业内部技术文档搜索**

- 员工搜索 "Kubernetes 1.28 升级步骤"
- 纯向量检索：返回各种 K8s 相关文档，可能包括 1.27、1.29 版本，相关但不精确。
- BM25：精确匹配 "1.28" 和 "升级"，返回的文档直接可用。
- 混合检索：向量召回语义相关的 K8s 文档 + BM25 精确命中版本号，合并后结果既全面又精确。

### 融合策略

- **RRF (Reciprocal Rank Fusion)：** 分别取两个检索结果的前 K 条，按排名倒数求和排序。简单有效，无需调权。
- **加权融合：** 对向量得分和 BM25 得分做归一化后加权求和。权重可根据场景调整（如代码搜索 BM25 权重更高）。

## 9. Function Calling Schema 优化 —— Description 的艺术

### 除了参数名和类型，description 是关键

模型选择工具和填充参数的准确率，很大程度上取决于 description 的质量。

### 优化策略

**1. 负向约束（Negative Constraints）：**

明确告诉模型"什么情况下不要用这个工具"。

```json
{
  "name": "search_user",
  "description": "按用户名精确搜索用户。注意：此接口只接受精确用户名，不支持模糊搜索或拼音搜索。如果需要模糊匹配，请使用 search_user_fuzzy。",
  "parameters": {
    "username": {
      "type": "string",
      "description": "用户的精确用户名（login name），不是显示名称（display name），不是邮箱前缀。例如：'zhangsan' 而非 '张三' 或 'zhangsan@company.com'。"
    }
  }
}
```

**2. 调用策略（Calling Strategy）：**

在 description 中嵌入"使用时机"和"调用顺序"。

```json
{
  "name": "create_issue",
  "description": "在 GitHub 仓库中创建 Issue。使用前必须先用 search_repo 确认仓库全名（格式 owner/repo）。如果仓库不存在，不要调用此工具，而是提示用户先创建仓库。",
  "parameters": {
    "repo": {
      "type": "string",
      "description": "仓库全名，格式为 owner/repo，例如 'facebook/react'。必须从 search_repo 的返回结果中获取，不要自行拼凑。"
    }
  }
}
```

**3. 给出正面和反面例子：**

```json
{
  "name": "format_date",
  "description": "将日期字符串格式化为 ISO 8601 标准。正确输入示例：'2024年1月5日'、'Jan 5, 2024'。不支持：相对日期如'昨天'、'上周三'——这些请用 resolve_relative_date 工具。",
  "parameters": {
    "date_string": {
      "type": "string",
      "description": "绝对日期字符串。正确：'2024-01-05'、'2024年1月5日'。错误：'昨天'、'上周'（这些不是绝对日期，会返回错误）。"
    }
  }
}
```

**4. 明确副作用（Side Effects）：**

```json
{
  "name": "send_email",
  "description": "发送邮件给指定收件人。警告：此操作会真实发送邮件，不可撤回。请在调用前务必向用户确认收件人和内容。仅当用户明确说'发送'或'确认发送'时才调用。"
}
```

## 10. 多工具场景下的高效工具选择策略

### 问题

当 Agent 拥有 50+ 甚至 200+ 个工具时，将所有工具描述全部塞入 system prompt 会导致：

- 上下文窗口严重浪费。
- 模型在选择工具时"分心"——见过多的无关工具描述会降低选择准确率。
- Token 成本线性增长。

### 方案一：分层过滤（Hierarchical Filtering）

**工具分组 → 意图识别 → 组内选择**

1. 将所有工具按功能分组（如：文件操作、网络请求、数据库、代码执行……）。
2. 第一轮（轻量 Prompt）：只给模型看**组名 + 组描述**，让模型判断当前任务属于哪个（些）组。
3. 第二轮：只将命中组内的工具完整 Schema 送入模型做精确选择。

**优点：** 大幅减少无关工具的干扰。  
**缺点：** 需要多一轮 LLM 调用，增加延迟。当任务跨组时（如既需要读文件又需要调 API），需要选中多个组。

### 方案二：基于检索的预选择（Retrieval-based Pre-selection）

**将工具描述 embedding → 语义检索 → 取 top-K**

1. 将所有工具的 name + description 预先 embedding 存入向量库。
2. 每轮对话时，将用户的 query + 当前上下文 embedding，检索语义最相关的 top-K（如 K=10）个工具。
3. 只将这 10 个工具的 Schema 送入模型。

**优点：** 一次检索即可，延迟低；检索质量随 embedding 模型进步而提升。  
**缺点：** 可能漏掉"语义上不太相关但实际需要"的工具（如用户说"帮我保存一下"，但 "save" 与 "write_file" 的语义相似度可能不如预期）。

### 方案三：混合方案（推荐）

```
第一层：检索预选 → 从 200 个工具中召回 top-15 候选
第二层：LLM 精选 → 将 15 个工具的 Schema 送入模型做最终选择
```

**关键设计细节：**

- 检索时对工具 description 做 query 扩展——不仅匹配用户原话，还匹配同义词和常见表述。
- 为关键/高频工具设置"始终可见"白名单，不受检索过滤影响（如 `task_complete`、`ask_user`）。
- 定期用实际调用日志微调检索模型——哪些工具在哪些上下文中最常被调用。

### 特殊场景

**当模型连续选择"错误工具"时：**

在下一轮 prompt 中将之前"选了但没用"的工具从候选集中移除，强制模型从剩余候选中重选。同时将"上次选择失败的原因"作为负反馈注入。
