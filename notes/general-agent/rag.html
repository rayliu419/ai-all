📖 RAG 检索增强生成

Retrieval-Augmented Generation — 企业落地标准方案

一、概述

RAG 的核心思路是：在向大模型提问前，先从外部知识库中检索相关文档，将检索到的内容作为上下文注入 prompt，让模型基于给定知识而非自身记忆来生成答案。

RAG 解决了纯 LLM 的两个根本缺陷：知识截止期（预训练数据有时间点）和 幻觉（模型在缺乏知识时胡编乱造）。企业落地时，RAG 是实现知识库问答最成熟也是成本最低的方案。

二、核心流程与机制

2.1 运作流程

检索 → 增强 → 生成

  1. 用户发起查询
  2. 在知识库中进行向量召回，找到语义最相关的文档片段
  3. 将检索到的文档注入 prompt（相当于注入相关上下文）
  4. LLM 基于 prompt（含检索结果）生成最终答案

2.2 与传统搜索的对比

阶段传统搜索RAG
召回倒排索引（关键词匹配）向量召回（语义匹配）
排序BM25 等统计方法语义排序 + Re-ranker

2.3 向量化原理

Transformer 的起点就是 Embedding（嵌入层 + 位置编码）。RAG 中的向量化是将一段文本压缩成一个 稠密向量（dense vector），用这个向量在语义空间中做最近邻搜索。

矩阵视角：Embedding 是一个 m × n 矩阵，m = 词汇表大小，n = 向量维度。每个词对应一个 n 维向量。

三、LLM vs Embedding 模型：本质区别

维度LLM（GPT / Llama）RAG Embedding 模型
训练目标预测下一个 Token（Causal LM）判断语义相似度（对比学习）
学到的内容词与词的局部搭配关系（如"公园"→"散步"）文本片段间的语义距离
输入/输出Token 序列 → 下一个 Token文本 → 稠密向量
对长文本各层不断变换向量，不擅长压缩为"死向量"专门训练以精确压缩语义

四、RAG 后训练（Post-Training）

RAG 的 Embedding 模型不是从零训练的，而是以 基座 LLM 的 Embedding 矩阵 作为初始化底子，再经过 三个阶段的特训：

4.1 训练路径

基座模型 → 对比学习微调 → 信息检索对齐 → 精准语义距离模型

4.2 三个阶段详解

阶段一：对比学习微调（Contrastive Fine-tuning）

用三元组数据对基座模型进行微调，把"会背书的脑子"训练成"懂语义距离的翻译官"。

三元组示例：

  • Anchor（锚点）："如何用手机开通网银？"
  • Positive（正样本，语义相同）："手机银行怎么激活？"
  • Negative（负样本，语义不同）："手机屏幕碎了怎么修？"（字面都有"手机"，但意思完全不同）

→ 训练目标：Positive 与 Anchor 的向量尽量近，Negative 与 Anchor 的向量尽量远。

阶段二：信息检索对齐（Retrieval Alignment）

让向量检索的结果匹配大模型的"胃口"和真实业务场景。分为三个子方向：

① 问答对齐（Query-Passage Alignment）

用户输入的是问题（"手机怎么开网银？"），但库里存的是答案/知识点（"用户需登录网银 APP 激活..."）。两者的词汇结构完全不同，需要专门训练让模型拉近问题和答案的语义距离。

② 排序与重排对齐（Re-rank Alignment）

向量初筛召回了 Top-10 文档，但前3个可能是废话，第5个才是核心答案。大模型有"长文本迷失"的问题。解决方案：

  • 训练一个专门的 Re-ranker（重排模型）
  • Re-ranker 不负责"大海捞针"，只对已捞出的 Top-10 做精细的"人类偏好排序"
  • 把含金量最高的文档排在最前面喂给 LLM

③ 任务感知对齐（Task-aware Alignment）

有时需要的不是"语义相似"，而是满足特定任务需求。例如搜索 "iPhone"，希望返回的不是原理介绍而是配套配件（如下拉推荐）。

五、注入上下文：Long-Context vs RAG

给大模型注入上下文有两种主流方式：长上下文直读 和 检索增强生成（RAG）。各有适用场景。

5.1 长上下文大模型直读（Long-Context）

把所有文档拼在用户问题前面，一次性喂给大模型。

维度说明
优势 全局关联强：Attention 机制能建立全文本的网状关联。适合跨章节对比（"对比 A 和 B 在 2025 年的预算差异"）或全局总结（"分析这 20 个合同的法律风险"）。
秒级更新：无需中间件，文件更新后下次请求直接发新版即可。
劣势 长文本迷失：模型对"开头"和"结尾"记得最牢，核心答案藏在第 45 万字时容易被忽略或产生幻觉。
适用低频、高价值的分析任务

5.2 检索增强生成（RAG）

先去向量数据库查出最相关的几段，和问题拼在一起喂给模型。

维度说明
优势 精准度高：提前把"针"捞出来摆在模型眼前，只喂几千字，模型不需要在几十万字里做选择题。
海量数据唯一解：当文档是几万个 Wiki 页面、几十万个客服话术时，任何大模型都装不下，RAG 是唯一解。
劣势 必须切片（Chunking）：答案分散在第 5 页和第 100 页时，检索器很难同时捞出，易导致"盲人摸象"。
更新成本高：文档更新需要重新切片、重新 Embedding、更新向量数据库。

5.3 切片问题的工程对策

当文章长度超过 Chunk Size（如 500 字），系统必须"咔嚓"一刀切开，导致 信息碎裂 和 语义断层。典型问题：指代不明（代词丢失）、上下文断层（因果/转折撕裂）、结构化数据被切烂。

① 滑动窗口与重叠计数（Overlap）

相邻 Chunk 之间保留一部分重叠内容，减少关键信息恰好落在切缝上的概率。

② 父子块检索（Parent-Child Retrieval）

准备阶段把文章切成很小的"子块"，同时保留所属的"父块"。向量数据库只拿子块做语义匹配（小块的语义更聚焦、检索更精准）。一旦子块被选中，顺藤摸瓜把父块（完整上下文）捞出来喂给模型。

③ 语义切片（Semantic Chunking）

用轻量级 Embedding 模型逐句扫描文章，计算相邻两句的语义相似度。发现语义剧烈变化（换话题/换章节）时才切一刀，确保同一主题始终留在同一个 Chunk 里。

④ 密集微调与文档摘要（Contextual Retrieval / Late Chunking）

Anthropic 等提出的后训练对齐思路：切片前先让大模型生成一段 100 字的全局背景摘要，然后强行贴在每一个切片的最前面，保证每个切片都带着全局上下文。

六、总结

RAG 的本质：用外挂知识库为大模型提供"开卷考试"环境，而非让模型做"闭卷考试"。

  • 核心价值：解决知识截止期 + 缓解幻觉，且不需要重新训练模型，成本极低
  • 与传统搜索的区别：从关键词匹配升级为语义匹配
  • Embedding 的关键洞察：LLM 的目标是预测下一个词，Embedding 的目标是度量语义距离——两者虽然共享基础语言理解，但训练方向完全不同
  • 后训练的三阶段：对比学习微调 → 信息检索对齐（问答/重排/任务感知）→ 精准 Embedding 模型
  • Re-ranker 的重要性：Top-10 的排序质量直接影响 LLM 的输出质量，重排是 RAG pipeline 中容易被忽视但极其关键的一环

RAG = 检索 + 增强 + 生成
不是让模型知道更多，而是让模型在回答时"翻书"
