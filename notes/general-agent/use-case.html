BOT Agent vs. Traditional Alert

特征 传统告警系统（Alerting） AI 智能体（AI Agent）
核心机制 基于硬编码规则和阈值 基于 LLM 的推理、规划和工具调用
触发方式 某个指标超过预设值（例如：CPU > 90%） 接收到复杂指令或检测到异常模式（例如：发布事件开始）
输出结果 简单的通知（邮件、短信、PagerDuty） 结构化的分析报告、RCA 和自主行动
思维能力 无。仅判断 True/False 有。能分解任务、制定计划、反思、自我修正
对数据的处理 仅关注单个指标是否超过阈值 能够综合分析多个指标（延迟、错误率、日志、基线）之间的复杂关系。这是核心，因为 LLM 具有核心知识
行动能力 主要是通知人来解决 可以调用 API 执行 rollback()、isolate_traffic() 或 auto_fix() 等复杂操作
主要目标 通知 SRE/DevOps 发生了什么问题 解决 为什么 发生问题，并主动处理

Review Agent

自由发挥原则：没定义在规则里面的也提评论，满足规则的也提。

GAN 模式

定义一个额外的 Debate Agent 来辩论。在 Q&A Agent 中也表现良好。

Prompt vs. Program

纯提示词构建的 Agent：稳定性和可靠性不够。

用 Agent SDK 构建的 Agent：稳定性和可靠性会强很多。

考虑知识库的帮助 Agent

  • 提示词方式：自主扩张搜索范围。某些情况下是好的，就像 explore subagent，不断尝试 grep 不同的词获取信息
  • 编程方式：在 Q&A Agent 中，搜索到不好的数据源。可以用 GAN subagent 验证，但编程方式更可靠，可以直接校验 MCP Tool 查找的 Wiki 数据源

因此，编程方式提供更高的可靠性。

💡 核心思路

Agent 相比传统告警系统的核心价值在于：从"告诉我出事了"升级为"告诉我为什么出事并处理它"。Review Agent 中 GAN 模式效果显著。编程构建的 Agent 比纯 Prompt 方式更可靠。
