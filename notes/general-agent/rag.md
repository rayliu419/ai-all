# 定义
- RAG 是 Retrieval-Augmented Generation 的缩写，中文通常翻译为 “检索增强生成”。
- 企业落地的标准方案。

## 为什么需要RAG
- 预训练知识有时间点。
- 没查到知识时的幻觉。
  - 其实我觉得是因为查到了以前的预训练知识。
  - 在知识库中如果完全没有知识时幻觉尤其严重。

## 实现机制
- 跟传统搜索类似。
  - 召回
    - 传统的倒排索引，RAG是向量召回。
  - 排序
    - 传统的是类似BM25。

## 运作流程
- 先在知识库检索一把，然后带着知识向model发请求。
- 相当于注入了相关上下文。

## 与LLM的向量化区别
- transformer中最开始就是embedding, 包括embedding + positional embedding。
- 也是一种一段话的表征向量抽取，包含语义。
- 也是乘以矩阵。
  - 矩阵是m * n, m是词汇表，n是向量维度。
- 大模型（LLM）的训练目标和 RAG 专用的 Embedding 模型的训练目标，有着本质的区别。
  - 大模型（如 GPT、Llama）在预训练阶段，它的目标是“预测下一个Token”（Causal Language Modeling）。这种训练带来的结果：这个矩阵里学到的是**“词与词的局部搭配关系”**。比如“公园”和“散步”因为经常前后脚出现，它们的向量会产生某种数学上的关联。但是，LLM 的每一层都在不断改变向量，它并不擅长把“一整段话”压缩成一个死向量。
  - 当我们要为RAG训练一个Embedding模型时，我们的核心需求变了：我们不需要模型去预测下一个词，而是需要模型能够判断“两段不同的话，语义是否相似”。为了实现这个目标，专门的Embedding模型通常会采用对比学习。

## RAG的后训练
- 所以，RAG 使用的 Embedding 矩阵，往往是拿 LLM 预训练好的矩阵作为底子（初始化），因为它已经具备了基本的语言逻辑；然后通过对比学习进行“特训”，把一个只会“背书接龙”的脑子，训练成一个能够精确定位“语义距离”的“超级翻译官”。
- Embedding的路径：基座模型 -> 对比学习微调（Contrastive Fine-tuning）-> 信息检索对齐（Retrieval Alignment）-> 变成能精准计算语义距离的Embedding模型。
  - 对比学习微调。3元组。
    - 对比学习不能只喂单条文本了，它需要成对的数据。最经典的是 (Anchor锚点, Positive正样本, Negative负样本) 三元组。
    - Anchor（原句）：“如何用手机开通网银？”
    - Positive（语义相同，说法不同）：“手机银行怎么激活？”
    - Negative（字面相似但语义不同，或完全无关）：“手机屏幕碎了怎么修？”（字面都有“手机”，但意思完全不同）
  - Positive尽量近，Negative尽量远。
  - 信息检索对齐。
    - 信息检索对齐，就是针对 Embedding 模型的“后训练对齐”，目的是让向量检索的结果完美匹配大模型的“胃口”和真实业务场景。
    - 问答对齐（Query-Passage Alignment）
      - 用户输入的是一个问题（“手机怎么开网银？”），但数据库里存的是答案/知识点（“用户需登录网银 APP，点击账户激活...”）。在语义空间里，问题和答案的词汇结构完全不同，普通对比学习很难把它们拉近。
    - 排序与重排对齐（Re-rank Alignment）
      - 向量数据库初筛（召回）出来的 Top-10 文档，可能前3个都是废话，第5个才是核心答案。大模型有“长文本迷失”的毛病（容易忽略中间的信息）。
      - 训练精细的Re-ranker（重排模型）。它不负责在大海里捞针，而是专门对捞出来的10根针进行精细的“人类偏好排序”，把含金量最高、最能解决问题的文档，强行排在最前面，喂给大模型。
    - 任务感知对齐（Task-aware Alignment）
      - 有时候，你希望向量检索出来的东西不是“语义相似”，而是满足特定任务。比如你输入一个商品“iPhone”，你希望检索出来的不是它的原理介绍，而是它的“配套配件”（如下拉推荐）。

## 注入上下文
- 长上下文大模型直读（Long-Context） 
  - 把所有文档拼在用户问题前面，一次性喂给大模型。
  - 大模型的Attention机制能够建立全文本的网状关联。如果你的问题需要跨章节对比（如：“对比 A 项目和 B 项目在 2025 年的预算差异”），或者需要全局总结（如：“分析这20 个合同里潜在的法律风险”），直读方式效果极好，因为全量信息都在模型的“大脑内存”里。
  - 虽然模型能放得下 100 万字，但存在“长文本迷失”（Lost in the Middle）现象。大模型往往对文本的“开头”和“结尾”记得最牢，如果核心答案藏在第 45 万字的某个角落，模型有一定概率会忽略它，或者在生成时产生幻觉。
  - 完全没有中间件。如果公司的制度文件更新了，你只需要在下一次请求时，把新的 PDF 连同问题一起发过去即可，实现“秒级更新”。
  - 低频、高价值的分析任务
- 检索增强生成（RAG）
  - 先去向量数据库查出最相关的几段，和问题拼在一起喂给模型。
  - RAG必须把文档“切片”（Chunking）。如果答案分散在第5页和第100页，且两段话字面语义不相似，RAG的检索器很难同时把它们捞出来，容易导致“盲人摸象”，漏掉关键上下文。
  - RAG提前把“针”捞出来，并摆在大模型眼前（只喂给模型几千字）。模型不需要在几十万字里做选择题，回答的精准度非常高。
  - 文档更新时，你需要重新做切片、重新跑Embedding 模型、更新向量数据库。如果切片策略没做好，可能会出现新老数据冲突。
  - 当你的文档总量不是几本 PDF，而是公司过去10年沉淀的几万个 Wiki 页面、几十万个客服话术时，任何大模型都装不下，RAG是**唯一解。**
  - 只要文章的长度超过了你设置的Chunk Size（切片大小，比如设为 500 字），系统就必须“咔嚓”一刀把它切成两半。这必然会导致你说的信息碎裂、语义断层。在实际工程中，粗暴的切片通常会带来以下三个严重问题
    - 指代不明（代词丢失）
    - 上下文断层（因果、转折撕裂）
    - 结构化数据（如表格、代码）被切烂
    - 工业界是如何解决这个问题的？
      - 滑动窗口与重叠计数（Overlap）
      - 父子块检索（Parent-Child Retrieval / 小块检索，大块喂给模型）
      - 在准备阶段，把文章切成很小的“子块”，同时也保留它们所属的更大的“父块”。向量数据库只拿子块去和用户的问题做语义匹配（因为小块的语义更聚焦，检索更精准）。一旦某个子块被选中，系统不把这个子块喂给模型，而是顺藤摸瓜，把它的父块（2000字的完整上下文）捞出来喂给大模型。
      - 语义切片（Semantic Chunking）。系统会用一个轻量级的 Embedding 模型逐句扫描文章，计算相邻两句话之间的语义相似度。 当发现某两句话之间的语义突然发生了剧烈变化（说明换话题、换章节了），系统才在这里切一刀。这样能确保同一个主题的内容始终留在同一个Chunk里。
      - 密集微调与文档摘要（Late Chunking / Contextual Retrieval）。这是 Anthropic（Claude 的母公司）等头部厂商提出的一种全新“后训练”对齐思路。在切片之前，先让大模型给整篇长文章生成一个100字的全局背景摘要。把这个全局摘要，强行贴在每一个切片的最前面。
